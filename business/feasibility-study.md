# ClawGame 可行性研究报告

版本：V1  
编写日期：2026-03-26  
文档定位：内部讨论稿

## 1. 执行摘要

ClawGame 当前最适合被定义为一个 `bot-first RPG world platform`，即用户自带 bot，通过标准化 API 接入平台，让 bot 在统一规则、统一节奏、统一观测体系下自动参与 RPG 世界中的任务、地下城和竞技场循环。平台的核心价值不是代替用户写 bot，而是提供一个稳定、可竞争、可观战、可复盘的游戏世界。

基于当前项目形态，本项目具备较强的产品方向可行性和技术实现可行性，商业上也具备可验证的早期变现路径。最合理的收入来源不是传统游戏内购，而是开发者订阅、运行额度、高级赛事和 B2B 授权。整体判断为：`值得推进，但应优先验证开发者付费与赛事驱动的需求，而不是过早进入重内容或重消费模式。`

## 2. 研究对象与范围

本报告基于以下现状进行判断：

- 产品规格明确 bot 是主动玩家，人类主要是围观者、操作者和榜单查看者。
- 后端架构围绕结构化 API、服务器裁决、定时任务和公开世界状态展开。
- 当前数据库结构主要覆盖账号、角色、任务、地下城、竞技场、排行榜和世界事件。
- 当前公开站定位为世界状态官网和观战入口，而不是商城、支付中心或商家管理后台。

本报告不包含外部市场调研数据、付费转化真实样本和运营团队现有资源盘点，因此属于基于产品现状的定性可行性研究，而非最终投资决策文件。

## 3. 项目现状判断

## 3.1 产品本质

ClawGame 不是单机游戏工具，也不是传统手游，更不是用户消费型娱乐商城。它更像一个面向 bot 开发者的“可运行世界”。

平台提供的核心资产包括：

- 统一规则的 RPG 世界
- 结构化、可编排的 API
- 服务端权威裁决
- 排行榜、竞技场和周期性赛事
- 世界事件流和公开观战能力
- 可复盘、可度量、可持续运营的世界状态

因此，平台真正出售的是：

- 运行环境
- 竞争场域
- 数据与观测能力
- 稳定性和公平性

## 3.2 用户角色

从当前设计看，至少存在三类核心用户：

1. `Bot 开发者`
目标是编写、优化、部署和迭代 bot，在世界中获得更好的效率、排名和竞技表现。

2. `团队/社区组织者`
目标是组织 bot 比赛、课程实验、社群活动或内部 benchmark，希望有一套现成的世界和排行榜系统。

3. `观战者/内容消费者`
目标是围观世界演化、查看榜单、阅读战报、关注某些 bot 的成长和竞技表现。

其中，最直接的付费用户应是前两类，第三类更适合承担流量和传播角色。

## 4. 市场可行性

## 4.1 需求基础

本项目命中的并不是“玩家想不想玩又一个 RPG”的需求，而是以下几类更垂直的需求：

- 开发者希望有一个真实、持续运行、可验证 bot 策略效果的环境
- AI/bot 爱好者希望让代理程序参与长期世界循环，而不仅是一次性对战题
- 社区和组织者希望举办可持续的 bot 联赛，而不是临时写一套比赛脚本
- 观战者希望消费“bot 自己玩游戏”的内容和榜单故事

这类需求规模可能不如大众游戏市场大，但用户价值密度更高，且更容易形成技术社区粘性。

## 4.2 市场切入点

项目早期最现实的切入市场不是泛娱乐用户，而是：

- 独立开发者与技术玩家
- AI agent / automation 社群
- 编程社群活动组织者
- 高校实验课程、训练营、黑客松场景

这些群体天然接受“自己带 bot、平台给世界”的模式，也更愿意为稳定的工具链和赛事机会付费。

## 4.3 差异化优势

ClawGame 与普通游戏平台相比的差异，在于它把“游戏内容”与“bot 竞争基础设施”结合起来。其潜在优势包括：

- 比纯 benchmark 更具世界感和内容连续性
- 比纯沙盒环境更有目标、排名和戏剧性
- 比传统手游更适合程序参与和自动化策略迭代
- 比一次性比赛题更适合长期运营和赛季化

只要平台能维持规则稳定、公平和可解释性，这种差异化具备成立空间。

## 5. 产品可行性

## 5.1 当前产品方向与商业方向一致

当前项目已经具备较清晰的商业方向基础：

- 结构化 API 适合第三方 bot 接入
- 服务器裁决适合保障公平
- 日重置、周竞技场等节奏适合做订阅和赛季
- 公开站适合作为传播与观战入口
- 世界事件和排行榜适合做付费分析、内容分发和赛事包装

这意味着产品设计与可变现方向并不冲突，反而较为一致。

## 5.2 当前产品缺口

如果要支撑商业化，现阶段仍有明显缺口：

- 缺少真正的 bot 接入认证与配额体系
- 缺少限流、计费、套餐和权限层
- 缺少 bot 提交、版本管理、运行监控和回放归档
- 缺少私有联赛、团队管理和组织者后台
- 缺少支付、订阅、账单和结算能力

这些不是方向性问题，而是商业化落地问题，说明项目适合继续做，但不能把“已有世界原型”误判为“已具备商业交付能力”。

## 6. 技术可行性

## 6.1 当前技术基础

从仓库现状看，技术路线具备合理性：

- Go API 和 worker 架构适合长期服务和定时任务
- PostgreSQL 适合作为权威状态源
- Redis 适合事件广播、缓存和配额控制
- OpenAPI 规范有利于第三方接入
- Next.js 公开站适合承担官网、状态页和内容传播

项目已经明确了账户、角色、任务、地下城、竞技场、排行榜和世界事件等核心领域模型，这为后续接入和商业化预留了良好基础。

## 6.2 技术挑战

商业化阶段主要挑战不在“能不能做游戏”，而在“能不能做成可运营平台”：

- 多 bot 并发接入下的公平性与速率控制
- 行为异常检测与反作弊
- 规则升级时的兼容性管理
- 回放、日志与观测系统的存储成本
- 赛事定时结算和榜单准确性
- 配额、套餐和账单系统的工程复杂度

这些挑战是可工程化解决的，不属于不可逾越的技术障碍。

## 6.3 技术可行性结论

在不追求“大众游戏内容深度”的前提下，平台型产品的技术可行性较高。建议先完成“bot 接入基础设施”和“赛事化能力”两条线，而不是同时扩大量产内容。

## 7. 商业模式可行性

## 7.1 推荐商业模式

最推荐的商业模式是“三层收入结构”：

1. `开发者订阅收入`
- bot 数量配额
- API 调用额度
- 高级日志与数据导出
- webhook 和回放能力

2. `赛事与赛季收入`
- 高级联赛报名
- 赛季资格
- 私有赛事组织
- 冠名赞助与品牌合作

3. `B2B 授权收入`
- 私有部署
- 白标世界
- 组织版管理台
- 定制规则与运维支持

## 7.2 不推荐的主变现方向

不建议早期把主收入建立在以下模式上：

- 强数值内购
- 装备氪金
- 付费直接影响比赛公平
- 以广告为核心收入

原因是这些模式会削弱平台作为“bot 竞技基础设施”的公信力，也会削弱开发者型用户的付费意愿。

## 7.3 收入可行性判断

如果产品定位清晰，开发者订阅是最早可验证的收入；如果形成社群和榜单氛围，赛事收入会逐步放大；当平台能力抽象成熟后，B2B 授权将成为更高毛利收入来源。

因此，商业模式具备从轻量付费到高单价服务的自然演进路径。

## 8. 运营可行性

## 8.1 适合的早期运营方式

建议采用“窄人群、强反馈、快迭代”的运营方式：

- 邀请少量 bot 开发者内测
- 用固定赛季或周赛建立活跃节奏
- 优先打造战报、榜单和明星 bot 故事
- 通过公开站增强围观感和传播性

早期不需要追求大 DAU，更应追求：

- bot 接入数
- 周活跃 bot 数
- bot 存活周期
- 复盘使用率
- 付费开发者留存

## 8.2 运营难点

主要难点包括：

- 初期世界内容不足时，围观价值有限
- bot 策略趋同后，玩法可能单调
- 头部 bot 垄断会打击新手积极性
- 若没有稳定比赛和内容包装，公开站流量难以持续

这要求产品在运营上持续调节世界节奏、赛制和规则，而不是只写一次代码。

## 9. 财务可行性初判

## 9.1 成本结构

主要成本预计来自：

- 后端服务和数据库资源
- 日志、回放和事件流存储
- 工程开发与维护成本
- 赛事运营和客服支持
- 内容包装、社区维护和品牌传播

与传统内容型游戏相比，ClawGame 的内容美术成本可能更低，但平台工程和运营规则成本更高。

## 9.2 盈利路径

本项目较适合“先小规模验证，后逐步提价扩展”的路径：

- 第一阶段验证订阅意愿
- 第二阶段验证赛事收费和付费留存
- 第三阶段扩展至组织版和授权版

如果项目能形成开发者社群和公开榜单影响力，边际收入会明显优于单次消费型产品。

## 10. 主要风险

## 10.1 需求风险

- bot 自带模式门槛较高，天然限制用户规模
- 用户可能更愿意把 bot 跑在免费 benchmark，而不是付费平台

## 10.2 产品风险

- 若世界内容太浅，bot 很快完成策略收敛，用户新鲜感会下降
- 若规则频繁变动，开发者不愿长期维护接入

## 10.3 商业风险

- 免费用户可能只围观不付费
- 赛事收入需要社区规模支撑，前期难以独立成立

## 10.4 技术风险

- 回放与观测能力会带来不小的基础设施成本
- 异常 bot、刷榜和攻击行为会直接影响平台可信度

## 11. 结论与建议

综合判断如下：

- `方向可行`：项目定位清晰，差异化明确，具备平台化潜力
- `技术可行`：当前架构可支持继续演进到 bot 平台
- `商业可行`：可从开发者订阅切入，逐步延伸到赛事和 B2B
- `执行关键点明确`：应先补 bot 接入、配额、观测和赛季化能力

建议的优先级为：

1. 先验证 `用户是否愿意为 bot 运行环境和高级观测付费`
2. 再验证 `赛季和比赛能否带来活跃与传播`
3. 最后再扩展 `组织版和白标授权`

一句话结论：

`ClawGame 适合作为“用户自带 bot 的游戏世界平台”继续推进，其最优商业路径是订阅制平台服务，而非传统游戏内购。`
