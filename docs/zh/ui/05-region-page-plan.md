# 区域页方案

最后更新：2026-04-10

状态：工作草案

## 1. 页面角色

区域页应该像一份地点档案。

它要帮助用户回答：

1. 这是什么地方
2. 这里现在发生了什么
3. 这里有哪些值得关注的系统和路径

## 2. 主要结构

建议顺序：

1. 共用顶部导航
2. 紧凑页面 Hero
3. 地点身份区
4. 本地动态流
5. 系统与设施
6. 行动与旅行网络
7. 相关地点

## 3. 页面 Hero

保留：

- 区域标题
- 简短介绍
- 紧凑统计

减弱：

- 重复全站框架感

Hero 负责建立地点身份，不应该像第二个首页。

## 4. 地点身份区

这应该是页面的第一个主内容区。

内容：

- 区域名称
- 类型
- 风险
- 旅行成本
- 一段强环境摘要

这一段应当像地点条目的开场介绍。

## 5. 本地动态流

这应该是页面里最“活”的部分。

作用：

- 展示 Bot 正在这里做什么
- 让区域有当下感，而不是静态设定

规则：

- 使用事件行，而不是厚重卡片
- 保持情报流易读
- 让角色和事件链接容易扫读

## 6. 系统与设施

这一段负责解释这个区域提供什么。

内容：

- 关键 NPC
- 设施
- 材料
- 成长用途

规则：

- 把相关世界功能组织在一起
- 不要变成孤立事实堆积

## 7. 行动与旅行网络

这一段负责解释 Bot 在这里能做什么，以及接下来去哪里。

内容：

- 可执行行动
- 关联建筑
- 旅行目的地

规则：

- 旅行要更像路线规划，不要只是原始链接列表

## 8. 当前实现中应调整的内容

- 保留强地点叙事开场
- 保留本地动态情报带
- 降低面板密度
- 让系统、行动、旅行按顺序更容易读
- 减少“每个子区块都像独立特殊面板”的感觉
