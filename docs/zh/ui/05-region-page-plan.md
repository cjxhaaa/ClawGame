# Regions 页方案

最后更新：2026-04-11

状态：工作草案

## 1. 页面角色

Regions 模块应同时支持“发现区域”和“查阅区域档案”。

它应帮助用户回答：

1. 应该先看哪个区域
2. 这个区域到底是什么地方
3. 这个区域关联了哪些副本、系统与路径

## 2. 主要结构

采用“两层结构”：

1. 区域列表页
2. 区域详情页

列表页负责浏览。
详情页负责像 wiki 档案一样查阅。

补充说明：

- 独立的世界日志面可以并入 Regions 档案流，而不必长期作为一级主入口存在

## 3. 区域列表页

用途：

- 成为可读的世界区域目录
- 支持 wiki 式发现
- 让用户先找到区域，再进入完整详情

建议结构：

1. 共用顶部导航
2. 紧凑 Regions Hero
3. 筛选与排序控件
4. 区域列表
5. 合并后的世界档案流
5. 可选的小型世界地图参考

建议筛选项：

- 区域类型
- 风险等级
- 成长阶段
- 是否包含副本
- 资源或功能标签

每条区域列表项应展示：

- 区域名
- 类型
- 风险
- 一句简短摘要
- 关键副本或设施提示
- 详情页入口

规则：

- 这个页应更像世界 wiki 索引，而不是数据看板

## 4. 区域详情页

推荐顺序：

1. 共用顶部导航
2. 紧凑页面 Hero
3. 左侧嵌入式地点名列表
4. 右侧区域档案
5. 下方独立的地区日志区块

## 5. 页面 Hero

保留：

- 区域标题
- 简短介绍
- 紧凑统计

减弱：

- 重复全站框架感

Hero 应建立地点身份，不要变成第二个首页。

## 6. 区域身份

这是第一个主要内容区。

内容：

- 区域名称
- 类型
- 风险
- 推荐阶段
- 旅行成本
- 一段强环境摘要

这一区块应该像地点条目的开场段落。

## 7. 区域概览

这里应像 wiki 开头说明。

内容：

- 这个区域最重要的特征
- 它在成长路径中的意义
- 它通常吸引哪类玩家或 Bot 活动

规则：

- 先帮助读者建立理解，再进入高密度系统细节

## 8. 嵌入式地点名列表

详情页左侧应保留一列地点名。

作用：

- 让切换不同区域变得直接
- 让用户始终停留在档案流里，而不是不断退回列表页

规则：

- 每一行都应是清晰可读的地点入口，而不是噪声很大的数据卡
- 当前区域需要被明确高亮
- 这个列表要嵌入页面结构里，不要再做成漂浮抽屉

## 9. 区域档案

这一块用来解释该区域是什么，以及它为什么重要。

内容：

- 区域身份
- 环境摘要
- 关键 NPC
- 主要设施
- 主要产出
- 成长用途
- 挑战摘要

规则：

- 让详情页整体更像一张可阅读的地点档案
- 从核心详情布局里移除动态情报流、旅行网络、建筑与动作控制面

## 10. 副本与挑战

这应当是区域详情页中最重要的区块之一。

内容：

- 关联副本
- 挑战类型
- 推荐战力或阶段
- 核心奖励或掉落方向
- 一句难度说明
- 是否适合支援玩法

规则：

- 副本信息应能被快速横向比较
- 不要把副本内容埋在低优先级琐碎信息后面

## 11. 资源与掉落

这一块用来解释区域能产出什么。

内容：

- 主要材料
- 代表性装备方向
- 特殊资源
- 这些产出有什么用途

规则：

- 让资源价值对人类用户可读，不要只像原始物品表

## 12. 地区日志

页面下方应有独立的地区日志区块。

内容：

- 当前区域最近公开事件
- 角色链接
- 相对时间

规则：

- 作为档案页下方的独立栏目存在
- 不要再和上方区域摘要混成一个实时情报面板

## 13. 相关链接

可选关联区块：

- 最近事件
- 相关 Bot
- 关联排行榜或竞技场入口

## 14. 当前实现中应调整的内容

- 保留强地点叙事开场
- 加入左侧嵌入式地点名列表，支持直接切换区域
- 降低面板密度
- 先建立完整的 Regions 索引页，再依赖详情页做深度查阅
- 把地区日志下沉为单独区块
- 从详情页移除动态情报流、旅行网络、建筑与动作面板
- 让系统、副本、资源、旅行按顺序更容易读
- 减少“每个子区块都像独立特殊面板”的感觉

## 15. 移动端适配

在移动端：

- 保持详情页 hero -> 身份 -> 概览 -> 当前动态 -> 设施 -> 副本 -> 资源 -> 旅行 的顺序
- 区域列表页让筛选器位于上方，列表变成清晰的纵向目录
- 优先保证副本区块可读，再考虑是否保留低优先级相关链接
- 设施与资源长列表优先折叠，不要先牺牲副本与当前动态
