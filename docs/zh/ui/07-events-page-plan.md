# 事件页方案

最后更新：2026-04-11

状态：范围收缩

当前方向：

- 独立的世界日志页不再是首选主浏览入口
- 全局事件阅读将并入 Regions 档案流
- `/events` 可以保留为兼容入口或跳转页，而事件详情页在被直接引用时仍可继续存在

## 1. 页面角色

事件页应当像“世界纪事”。

它应回答：

1. 最近发生了什么
2. 当前主导世界的是什么类型的行为
3. 哪些区域现在最热

优先级说明：

- 这个页面很重要，但在主浏览顺序上低于 Regions 和 Arena

## 2. 主要结构

推荐顺序：

1. 共用顶部导航
2. 紧凑页面 Hero
3. 事件时间线头部
4. 主事件流
5. 侧边上下文栏

## 3. 事件时间线头部

内容：

- 分类控件
- 当前过滤器
- 如果存在，则显示地区范围
- 分页或每页数量控制
- 一句说明事件流行为

规则：

- 过滤器很重要，但真正的主角应当是时间线本身
- 事件量已经足够大，所以分类与分页必须作为一等需求来设计

## 4. 主事件流

这是页面的核心。

每行应展示：

- 事件类型
- 摘要
- 行动者
- 区域
- 相对时间

规则：

- 事件行要更像世界时间线，不要像社交卡片

## 5. 侧边上下文栏

可选内容：

- 区域热度
- 竞技场状态
- 当前事件类型分布

规则：

- 它应辅助用户理解时间线，而不是分散注意力

## 6. 当前实现中应调整的内容

- 保留无限加载和时间顺序逻辑
- 保留地区范围支持
- 强化分类清晰度与分页行为
- 强化“单条世界时间线”的感觉
- 减少每条事件额外的包裹感

## 7. 移动端适配

在移动端：

- 保持 hero -> 时间线头部 -> 事件流 -> 上下文摘要 的顺序
- 侧边上下文统一合并为消息流下方的紧凑区块
- 优先保留连续事件行，再考虑是否保留区域热度等可选摘要
- 筛选器可以堆叠在事件流上方，但不能抢过时间线本身的视觉主导权
