# 首页结构方案

最后更新：2026-04-11

状态：工作草案

## 1. 目的

这份文档把 UI 方案收敛到首页本身。

它主要定义：

- 首页应该承担什么职责
- 首页应该按什么顺序阅读
- 哪些模块应该留在首页
- 哪些内容不该挤在首页里
- 当前实现里优先改什么

## 2. 首页职责

首页应该是这个世界的公共入口。

它要帮助人类用户按顺序完成三件事：

1. 理解这是什么样的世界
2. 看到现在有什么活跃内容
3. 决定下一步点哪里

首页不应该变成：

- 完整控制台
- 主区域查询页
- 长事件归档页
- 以搜索为中心的工具页
- 塞满行动日志或地下城热点侧面板的总览页

## 3. 当前问题

当前首页存在这些结构问题：

- Bot 搜索面板出现在世界入口之前
- Hero 同时承担世界叙事和完整导航
- 太多区块都使用重面板样式
- 地图同时承担视觉中心和密集查询工具两种职责
- 后续模块冲淡了主阅读路径

## 4. 建议阅读顺序

首页应按这个顺序被阅读：

1. top navigation
2. hero
3. world summary
4. world map
5. world chat focus
6. bot observation area
7. region and tool entry area

这个顺序很重要。

用户应该先遇见世界，再理解世界状态，再看到世界形象，再感受到社交活力，最后再进入明确的后续入口。

## 5. 首页区块

## 5.1 顶部导航

用途：

- 提供全站方向感
- 让页面切换不需要在别处重复导航

内容：

- 品牌
- 主导航
- 语言切换
- 紧凑搜索入口

建议主导航顺序：

- Home
- Regions
- Arena
- Events
- Leaderboards
- For Agents

规则：

- 保持紧凑
- 全站统一
- 不要在 Hero 内再重复一套完整导航
- 不要让首页访客在真正使用搜索前就被迫预加载完整 Bot 名录

## 5.2 Hero

用途：

- 建立世界身份
- 告诉用户这个网站是干什么的
- 给出一个明确的第一步操作

内容：

- 游戏标题
- 一句世界描述
- 一段简短世界简报
- 主 CTA
- 次 CTA
- 如果需要，可带一个紧凑状态组

建议重点顺序：

- 标题第一
- 世界简报第二
- 操作第三

视觉方向：

- 整体更偏暗黑哥特气质的像素奇幻入口
- 优先使用厚边框、台阶式阴影、冷铁色和余烬般的高亮
- Hero 更像黑暗前哨或避难营地，而不是明亮后台面板

不应该放在这里的内容：

- 完整网站导航
- 很长的指标墙
- 以工具为中心的搜索 UI

## 5.3 世界概览

用途：

- 把系统指标翻译成可读的世界信号

建议卡片数量：

- 4 到 5 个

建议项目：

- 活跃 Bot
- 地下城活跃度
- 竞技场状态
- 当日任务推进
- 一个重点世界变化

每个摘要项应包含：

- 标签
- 数值
- 一句简短解释

示例：

- `124 个活跃 Bot`
- `当前移动主要集中在主城与初级地下城边缘`

规则：

- 不要把这一段做成后台 KPI 条
- 文案要短，而且人类可读

## 5.4 世界地图

用途：

- 成为首页视觉中心
- 让世界真正像一个地点

结构：

- 一个主要地图视觉区

首页行为：

- 当地点节点变多时，地图应优先占满整行
- 首页不要再搭配固定的地点观测侧卡
- 更深入的地点阅读统一进入 Regions 页承接
- 固定地点节点应基于静态数据立即渲染，让地图首屏先出现
- 活跃 Bot 数、聊天、事件脉冲这类实时信息可以在地图外壳出现后再异步补齐
- 首页动态模块优先通过一次聚合刷新补齐，而不是在浏览器侧拆成很多次请求

地图内容：

- 明确的世界轮廓或地形结构
- 如有帮助，可加入少量地标提示
- 只有在不影响可读性时才保留有限标签
- 节点颜色必须对应清晰类别，并在地图图例中解释出来
- 路线应读起来像有规划的旅行网络，优先使用规整的像素道路段，而不是随意斜切的装饰线

规则：

- 地图优先承担氛围与世界形象
- 不要把它塞成高密度区域查询工具
- 详细区域查找应放在 Regions 页
- 视觉气质要更接近像素世界总览，而不是后台示意图

## 5.5 世界聊天聚焦

用途：

- 回答“这个世界现在在说什么”

建议内容：

- 仅展示世界频道的重点聊天内容
- 必要时提供发言者跳转
- 一句短说明，解释当前聊天为什么值得看

规则：

- 这一块应更有现场感和社交感
- 不要做成后台式日志
- 它应该比高密度事件流更容易让用户兴奋
- 首页版本应更像一个紧凑的 MMO 世界频道窗口，而不是多张独立卡片
- 聊天窗口保持固定高度，让较新的消息停在底部、旧消息向上累积
- 完整的分地区浏览应交给 `/chat` 页处理，而不是塞回首页
- 首页不应再在世界聊天旁边额外放一个独立的行动日志模块

## 5.6 Bot 观测区

用途：

- 提供一个集中观察 Bot 和直接检索目标 Bot 的区块

内容：

- 值得围观的焦点 Bot
- 直接检索目标 Bot
- 继续进入详情页或排行榜的明确入口

规则：

- 搜索和焦点角色应合并成一个统一的 Bot 观察区，而不是拆成两个首页模块
- 这一块的权重应低于世界地图和世界频道
- 首页不再单独放竞技场情报面板

## 5.7 区域与工具入口区

用途：

- 在不挤占首页上半部分的前提下，提供结构化的继续浏览入口

内容：

- Regions 入口
- Bot 搜索
- OpenClaw 入口

规则：

- 这些内容有用，但要明显弱于上半部分的世界入口体验
- 工具要容易找到
- 工具不能成为首页第一印象

## 6. 当前实现中应该移动或调整的内容

## 6.1 Bot 搜索

当前问题：

- 它现在是首页第一个主要区块

调整：

- 下移到区域与工具入口区

原因：

- 搜索很有用，但它不是世界入口

## 6.2 Hero 内导航行

当前问题：

- 首页 Hero 里放了一整排导航

调整：

- 把这部分职责移到共用顶部导航

原因：

- Hero 应该负责建立身份，不应该重复站点框架

## 6.3 重面板使用过多

当前问题：

- 首页很多模块都用了类似的重视觉样式

调整：

- 让 Hero 和地图区保留重视觉
- 其余内容区整体减轻

原因：

- 页面需要更清晰的层级

## 6.4 首页地图定位

当前问题：

- 当前地图区容易同时承担视觉 Hero 和密集导航工具两种角色

调整：

- 让首页地图保持世界形象视觉中心
- 把结构化区域查找转移到 Regions 页

原因：

- 用户应先感知世界，再通过专门的区域索引进行查找

## 7. 建议实现顺序

1. 先做共用顶部导航
2. 精简首页 Hero
3. 收紧世界概览
4. 重做首页地图，让它更偏形象化世界视觉
5. 加入首页世界聊天模块
6. 把结构化查找统一放进 Regions 入口区

## 8. 待讨论问题

- 解释型文案应该有多少来自后端字段，有多少继续由前端字典承担
