### 13.4 Quest 服务函数

- `EnsureDailyQuestBoard(characterID, businessDate) -> QuestBoard`
- `ListQuests(characterID) -> QuestBoardView`
- `GetQuestRuntime(characterID, questID) -> QuestRuntime`
- `AcceptQuest(characterID, questID) -> Quest`
- `GenerateBoardQuests(characterID, businessDate, boardProfile) -> []Quest`
- `BuildQuestFromTemplate(characterID, templateID, seed) -> Quest`
- `ApplyQuestTrigger(characterID, trigger) -> []QuestProgressChange`
- `ApplyQuestChoice(characterID, questID, choiceKey, payload) -> QuestRuntime`
- `AdvanceQuestInteraction(characterID, questID, interaction) -> QuestRuntime`
- `CompleteQuestIfEligible(questID) -> Quest`
- `SubmitQuest(characterID, questID) -> QuestSubmissionResult`
- `RerollQuestBoard(characterID) -> QuestBoardView`

补充说明：

- 当前任务模板目录应加载自 `apps/api/internal/quests/configs/`
- 当引入新的任务模板字段时，应先补英文后端文档，再同步中文，再改代码
- `GenerateBoardQuests` 负责按 `3 normal + 2 hard + 1 nightmare` 组装任务板
- `BuildQuestFromTemplate` 负责把模板实例化为当天的具体任务
- `ApplyQuestTrigger` 是统一推进入口，travel / field / dungeon / building 都应走它
- `ApplyQuestChoice` 用来承接 `nightmare` 任务里的推理分支
- `AdvanceQuestInteraction` 用来处理显式任务步骤，不把所有逻辑都硬塞进通用动作层

#### 13.4.1 Quest YAML 目录布局

建议目录结构：

```text
apps/api/internal/quests/configs/
  /daily
    010-kill-region-enemies.yaml
    020-collect-materials.yaml
    030-deliver-supplies-normal.yaml
    040-deliver-supplies-hard.yaml
    050-investigate-anomaly.yaml
    060-clear-dungeon-nightmare.yaml
  /supplemental
    010-curio-followup-delivery.yaml
    020-kill-dungeon-elite.yaml
```

每个 YAML 文件应描述：

- 任务身份字段：`template_type`、`difficulty`、`flow_kind`、`rarity`
- 生成元信息：任务池、排序、daily/supplemental 归属
- 当前运行时会直接消费的展示内容：`title`、`description`
- 目标与奖励占位
- runtime 步骤图：`initial_step_key`、`completion_step_key`、`steps`
- 显式交互定义
- 显式 choice 定义
- 推进规则：trigger type、progress source、前置条件

交付建议：

- 新增一个普通任务，优先应只是新增一个 YAML 文件
- 只有在引入全新机制时，才需要继续改 Go 代码

## 14. Worker Jobs

这些是必须存在的定时或后台任务。

### 14.1 每日重置任务

调度：

- 每天 `04:00 Asia/Shanghai`

职责：

- 重算 `character_daily_limits`
- 过期前一日任务板
- 为活跃角色生成新的任务板
- 产出汇总事件或管理指标

### 14.2 竞技场生命周期任务

#### 报名窗口任务

调度：

- 每天 `04:00` 重置后到 `09:00` 资格锁定前

职责：

- 创建下一个 `signup_open` 状态的赛事行

#### 工作日积分赛任务

职责：

- 周一到周五开放积分挑战
- 为每个角色重置当日 `3` 次免费挑战次数
- 允许购买额外挑战次数，最多 `10` 次
- 维护积分候选池，确保每个角色都能看到积分相近分段中的 `5` 名随机对手

#### 周五资格锁定任务

职责：

- 周五收盘后冻结本周积分榜
- 取前 `64` 名进入周六淘汰赛
- 若真实参赛者不足 `64`，补入 NPC
- 创建周六对阵表
- 开放冠军押注市场与周六单场胜负市场

#### 周六淘汰赛结算任务

职责：

- 自动推进周六 top-64 淘汰赛各轮
- 为每场对局持久化 battle log 与紧凑 battle report
- 结算已结束的单场胜负押注
- 为每个角色写入竞技场历史或等价的可查询个人历史索引
- 产生 `arena.match_resolved`

#### 周日收尾任务

职责：

- 产出最终排名
- 结算冠军押注
- 为 `top 32`、`top 16`、`top 8`、`top 4`、`runner-up`、`champion` 发放一周有效的竞技场称号
- 覆盖旧称号并刷新持续时间
- 写入排行榜快照
- 产生 `arena.completed`

### 14.3 清理任务

- 清理过期幂等键
- 清理已撤销的认证 session
- 过期长期悬挂的地下城运行
- 如果未来增加策略，再压缩或归档旧的公共事件

## 15. 横切校验规则

### 15.1 角色创建

- V1 每个账号只允许有一个角色
- `weapon_style` 必须与所选 `class` 匹配

### 15.2 旅行

- 当前存在未结算的主动战斗回合时不能旅行
- 目标区域必须已激活且已解锁

### 15.3 装备

- 物品所有者必须等于当前角色
- 装备槽位必须与物品槽位匹配
- 平民会跳过职业 / 武器流派限制校验
- 非平民仍需满足职业 / 武器流派限制

### 15.4 任务完成

- 提交时必须校验每日上限
- 同一个 source event 对任务推进应保持幂等
- 同一个 source trigger 不能让同一任务推进两次
- 多步任务的推进必须可重放且可恢复

### 15.5 地下城运行

- 同一角色不允许并行多个 active run
- run owner 必须等于请求发起人
- 已结束的运行除了管理修复外不可再变更

### 15.6 竞技场

- 同一赛事同一角色只能报名一次
- 报名窗口关闭后不能再报名
- 对阵种子必须是确定性的

## 16. 事件契约

每个会影响公共读模型的重要领域变更都必须写入一条 `world_events`。

### 16.1 必要 payload 约定

所有事件 payload 都应包含足够上下文，使 UI 卡片无需额外昂贵查询就能直接渲染。

`quest.submitted` payload 示例：

```json
{
  "quest_id": "quest_01JV...",
  "quest_title": "Clear 6 Forest Enemies",
  "reward_gold": 120,
  "reward_reputation": 20,
  "new_reputation_total": 245
}
```

`dungeon.cleared` payload 示例：

```json
{
  "run_id": "run_01JV...",
  "dungeon_id": "ancient_catacomb",
  "reward_gold": 260,
  "item_drop_catalog_ids": [
    "priest_ring_rare_001"
  ]
}
```

`arena.match_resolved` payload 示例：

```json
{
  "tournament_id": "tourn_01JV...",
  "match_id": "match_01JV...",
  "round_number": 2,
  "winner_character_id": "char_01JV...",
  "loser_character_id": "char_01JV...other",
  "summary": "bot-alpha defeated bot-beta in round 2."
}
```

## 19. 后端完成定义

当满足以下条件时，可视为 V1 后端定义完整：

- 所有枚举都已稳定并写入文档
- 所有主表都已具备迁移
- 所有公开接口与 Bot 接口都已有请求 / 响应结构定义
- worker 任务覆盖全部定时玩法功能
- 每个重要领域变更都包含 world event 产出
- 官网可以完全依赖公开 API 渲染，而不需要 DB 直连捷径
