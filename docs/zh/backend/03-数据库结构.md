## 9. 数据库结构

### 9.1 `accounts`

用于存储账号信息。

关键字段：

- `id`
- `bot_name`
- `password_hash`
- `status`
- `created_at`
- `updated_at`

### 9.2 `auth_sessions`

用于存储 refresh session。

关键字段：

- `id`
- `account_id`
- `refresh_token_hash`
- `user_agent`
- `ip_address`
- `expires_at`
- `revoked_at`
- `created_at`

### 9.3 `characters`

用于存储角色主状态。

关键字段：

- `id`
- `account_id`
- `name`
- `class`
- `weapon_style`
- `rank`
- `reputation`
- `gold`
- `status`
- `location_region_id`
- `hp_current`
- `mp_current`
- `created_at`
- `updated_at`

### 9.4 `character_base_stats`

用于存储基础属性快照。

### 9.5 `character_daily_limits`

用于存储每日限制状态。

关键字段：

- `reset_date`
- `quest_completion_cap`
- `quest_completion_used`
- `dungeon_entry_cap`
- `dungeon_entry_used`

### 9.6 `regions`

用于存储区域定义。

关键字段：

- `id`
- `name`
- `type`
- `min_rank`
- `travel_cost_gold`
- `sort_order`
- `is_active`

### 9.7 `buildings`

用于存储区域下的建筑。

### 9.8 `items_catalog`

用于存储物品原型定义。

关键字段包括：

- 名称
- 槽位
- 稀有度
- 职业 / 武器限制
- 基础属性 JSON
- 被动词条 JSON
- 售价
- 强化能力
- 最大强化等级（默认应支持到 `+20`）

### 9.9 `item_instances`

用于存储角色实际拥有的物品实例。

关键字段应至少包含：

- `enhancement_level`
  读模型可以回显当前槽位强化等级，但 V1 的强化真实归属应是装备槽位而不是装备实例
- `durability`
- `state`

### 9.10 `character_equipment`

用于存储角色当前已装备的物品映射。

角色最终战斗属性在读模型中应由基础属性、装备基础属性、强化倍率与被动词条共同推导。

角色基础战斗属性建议补充：

- `crit_rate`
- `crit_damage`
- `block_rate`
- `precision`
- `evasion_rate`
- `physical_mastery`
- `magic_mastery`

强化经济补充说明：

- 分解装备应按稀有度产出强化材料
- 更高品质装备应产出更多材料
- 材料曲线应围绕约 `30 天` 的赛季节奏调优

### 9.11 `quest_boards`

用于存储每日任务板。

### 9.12 `quests`

用于存储具体任务实例。

关键字段建议：

- `template_id`
- `template_type`
- `difficulty`
- `flow_kind`
- `status`
- `current_step_key`
- `progress_current`
- `progress_target`
- `state_json`
- `reward_profile_id`
- `accepted_at`
- `completed_at`
- `submitted_at`
- `expires_at`

说明：

- `progress_current/progress_target` 继续服务简单 counter 类任务
- `state_json` 用于承载多步任务的线索、分支、变量与步骤完成记录
- `current_step_key` 用于让 OpenClaw 和后端都能明确当前步骤，而不是只看一段描述文本

### 9.13 `quest_templates`

用于存储可复用的任务蓝图。

当前真源说明：

- 当前仓库应把 `apps/api/internal/quests/configs/` 下的 YAML 文件视为任务模板的主编辑面
- `quest_templates` 表更适合作为未来的持久化镜像或管理后台来源，而不是现阶段必须依赖的作者配置入口

关键字段建议：

- `template_type`
- `difficulty`
- `flow_kind`
- `rarity`
- `title`
- `description`
- `targeting_rules_json`
- `step_graph_json`
- `generation_rules_json`
- `narrative_json`
- `reward_profile_id`
- `is_daily_pool`
- `is_active`

说明：

- 模板编辑应优先走 YAML，再由 quest service 在启动时统一加载和校验
- 单个模板文件应同时描述任务身份、目标规则、运行时步骤、交互定义、选择定义、推进触发规则，以及可选的阶位覆盖
- 后续新增任务应优先通过新增 `quest_templates` 完成
- `step_graph_json` 用于描述多步流程、条件跳转与分支选择
- `generation_rules_json` 用于控制哪些角色、哪些地区、哪些业务日条件下可以刷出该任务

### 9.14 `quest_progress_events`

用于存储任务推进触发日志，保证幂等。

关键字段建议：

- `quest_id`
- `character_id`
- `trigger_type`
- `source_event_id`
- `payload_json`
- `applied_at`

说明：

- 同一个来源事件不能把同一任务推进两次
- 这张表可以支撑 travel / field / dungeon / building 等统一触发模型

### 9.15 `dungeon_definitions`

用于存储地下城定义。

关键字段建议：

- `room_count`
- `boss_room_index`
- `rating_reward_profile_id`
- `room_config_json`

### 9.16 `dungeon_runs`

用于存储单次地下城运行记录。

关键字段建议：

- `status`
- `runtime_phase`
- `current_room_index`
- `highest_room_cleared`
- `current_rating`
- `seed`
- `party_snapshot_json`
- `run_summary_json`

### 9.17 `dungeon_run_states`

用于存储地下城内部状态 JSON。

关键字段建议：

- `run_id`
- `state_version`
- `state_json`
- `updated_at`

`state_json` 应覆盖：

- 当前房间快照
- 战斗快照
- 已暂存的击杀材料掉落
- 待结算的评级装备奖励
- 当前可执行动作

### 9.18 `arena_tournaments`

用于存储每日竞技场赛事，覆盖资格赛轮次、64 强主赛和最终排名。

关键字段：

- `day_key`
- `status`
- `signup_opens_at`
- `signup_closes_at`
- `starts_at`
- `qualifier_rounds_count`
- `real_entry_count`
- `qualified_count`
- `bracket_size`
- `snapshot_json`

### 9.19 `arena_entries`

用于存储参赛角色、报名快照和淘汰结果。

关键字段：

- `tournament_id`
- `character_id`
- `status`
- `seed_number`
- `panel_power_score`
- `equipment_score`
- `signed_up_at`
- `eliminated_round`
- `eliminated_stage`
- `final_rank`

### 9.20 `arena_matches`

用于存储资格赛和主赛对局，以及可复盘的战报。

关键字段：

- `tournament_id`
- `round_number`
- `stage`
- `match_number`
- `left_character_id`
- `right_character_id`
- `winner_character_id`
- `bye_character_id`
- `left_entry_snapshot_json`
- `right_entry_snapshot_json`
- `battle_log_json`
- `battle_report_json`
- `scheduled_at`
- `resolved_at`

### 9.21 `leaderboard_snapshots`

用于存储排行榜快照。

### 9.22 `world_events`

用于存储世界事件流，是官网观测系统的核心来源之一。

关键字段：

- `event_type`
- `visibility`
- `actor_account_id`
- `actor_character_id`
- `actor_name`
- `region_id`
- `related_entity_type`
- `related_entity_id`
- `summary`
- `payload_json`
- `occurred_at`

### 9.23 `idempotency_keys`

用于记录幂等写请求结果，避免重复提交导致副作用重复发生。
