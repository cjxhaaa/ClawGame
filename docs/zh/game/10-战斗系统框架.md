# 战斗系统框架

## 1. 设计目标

这份文档定义后续副本怪物系统要建立在其上的完整战斗基础。

目标：

- 战斗要足够确定，让 Bot 可以围绕它做稳定规划
- 同时保留一定变化，让敌人和副本机制有身份感
- 所有重要决策都要通过结构化日志暴露出来
- 第一阶段就支持最多 `4` 名 Bot 组队下副本
- 战斗中角色自动出手，但出手逻辑必须可解释、可配置
- 与前面已经定义好的赛季属性系统和装备系统保持一致

## 2. 战斗模型

V1 战斗模型是：

- 完全由服务器裁定
- 完全回合制
- 菜单式动作，不做站位走位型动作战斗
- 除了明确声明的随机点外，尽可能确定
- 逐步记录日志，便于调试、观测和 Bot 学习

每场战斗统一按以下阶段推进：

1. 战斗初始化
2. 回合开始
3. 单位轮流行动
4. 回合结束结算
5. 胜负结算

## 3. 战斗范围

第一版需要支持以下战斗范围：

- `field_skirmish`
  - 用于任务、材料、经验循环的短战斗
- `elite_encounter`
  - 更难的单波或双波精英战
- `dungeon_wave`
  - 标准副本房间战斗
- `boss_phase`
  - 命名 Boss 战或其某一阶段
- `arena_duel`
  - 后续 Bot 对 Bot 的竞技场模式

底层战斗引擎应当统一，不同之处主要体现在：

- 参战模板
- 波次数
- 奖励包
- AI 脚本
- 失败条件

## 4. 战斗实体模型

每个战斗实体建议暴露以下字段：

- `entity_id`
- `team_id`
- `entity_type`
  - `player`
  - `monster`
  - `summon`
- `template_id`
- `name`
- `level`
- `rank`
- `stats`
- `current_hp`
- `shield_value`
- `status_effects`
- `cooldowns`
- `equipped_skills`
- `alive`

建议的属性载荷：

- `max_hp`
- `physical_attack`
- `magic_attack`
- `physical_defense`
- `magic_defense`
- `speed`
- `healing_power`
- 未来可选扩展：
  - `accuracy`
  - `status_mastery`
  - `status_resistance`

## 5. 战斗开始规则

战斗开始时：

- 所有实体从模板与运行时修正生成
- 当前 HP 默认设为满值，除非遭遇定义另有说明
- 先应用入场被动与套装效果
- 锁定本次副本的队伍成员与其 4 技能装配
- 可以提前计算并记录首回合行动顺序预览
- 向观测系统与回放系统发出战斗元信息

必须包含的战斗元信息：

- `battle_id`
- `battle_type`
- `encounter_id`
- `region_id`
- 如果有则包含 `dungeon_id`
- `party_size`
- `wave_index`
- `started_at`

## 6. 回合结构

每个回合按以下顺序执行：

1. 回合开始效果
2. 计算行动顺序
3. 所有存活单位各行动一次
4. 回合结束效果
5. 胜负检测

副本回合上限规则：

- 单场副本战斗总回合数上限为 `10`
- 如果在第 `10` 回合结束时仍有敌方存活，则本次副本判定失败
- 这种“超回合失败”不消耗每日副本次数
- 非超回合失败仍按副本模式决定是否消耗次数

回合开始效果包括：

- 处理在回合开始时失效的限制效果
- 结算标注为“回合开始触发”的恢复或延迟效果
- 输出回合开头日志

回合结束效果包括：

- `poison`、`burn` 伤害
- `regen` 治疗
- 持续回合数递减
- 到期状态清理

## 7. 行动顺序规则

行动顺序按以下优先级决定：

1. `speed` 更高者优先
2. `entity_id` 更小者优先

规则：

- 每回合开始重新计算顺序
- 改变 `speed` 的效果会影响下一回合排序
- 眩晕、冻结类跳回合不会把单位移出顺序表，只是到它时直接消耗该回合行动

## 8. 动作类型

V1 支持以下动作家族：

- `basic_attack`
- `skill_attack`
- `heal`
- `shield`
- `cleanse`
- `buff`
- `debuff`
- `summon`
- `wait`
- `escape`

第一版副本内容中，`summon` 与 `escape` 可以先做到引擎支持、内容暂不启用。

动作选择规则：

- 角色不会等待玩家手动输入
- 所有玩家侧单位都按自动战斗逻辑行动
- 自动战斗从“本次携带的 4 个技能 + 普通攻击”中选一个合法动作

## 9. 目标选择规则

目标类型建议包括：

- `self`
- `single_ally`
- `single_enemy`
- `all_allies`
- `all_enemies`
- `front_enemy`
- `lowest_hp_ally`
- `highest_threat_enemy`
- `random_enemy`

V1 目标设计原则：

- 玩家技能不要设计成含糊不清的目标逻辑
- 怪物 AI 可以使用简单但明确的脚本优先级
- 任何随机目标选择都必须写入日志

## 10. 仇恨与目标优先级

为了让副本怪物行为更清晰，战斗系统建议内置轻量仇恨模型。

每个怪物都会记录对玩家侧实体的仇恨表。

仇恨产生规则：

- 直接造成伤害：按最终伤害 `1.0x`
- 直接治疗：按有效治疗 `0.45x`
- 施加护盾：按护盾值 `0.35x`
- 嘲讽技能：注入固定高额仇恨

怪物默认选目标逻辑：

1. 若脚本或嘲讽强制指定目标，则优先它
2. 否则攻击当前最高仇恨的存活目标
3. 若该怪物是处决型逻辑，则优先当前生命最低目标
4. 若没有其他条件，则随机选择一个存活目标

设计原因：

- 这样后续坦克型构筑才有意义
- 也能让怪物行为可解释，而不是纯随机乱打

## 11. 命中、随机与波动

为了让 Bot 友好，战斗随机性要严格受控。

规则：

- 普通攻击命中率固定 `100%`
- 技能如果不是必定命中，必须显式定义命中率
- 第一版不使用隐藏闪避属性
- 普攻没有随机暴击
- 伤害浮动固定在 `+/- 5%`
- 所有随机点都必须记录在日志里

允许存在的随机点：

- 命中或未命中
- 状态施加是否成功
- 随机选目标
- 伤害浮动
- 战斗后的掉落判定

## 12. 伤害与治疗公式族

物理伤害：

`final_physical = max(1, (skill_power + actor.physical_attack * atk_ratio) - target.physical_defense * def_ratio)`

魔法伤害：

`final_magic = max(1, (skill_power + actor.magic_attack * atk_ratio) - target.magic_defense * def_ratio)`

治疗：

`final_heal = max(1, skill_power + actor.healing_power * heal_ratio)`

护盾：

`final_shield = max(1, skill_power + actor.healing_power * shield_ratio)`

伤害处理顺序：

1. 先算原始效果值
2. 再应用乘区增伤或减伤
3. 护盾优先吸收伤害
4. 剩余伤害再扣 HP
5. 最后处理命中后的附加状态

## 13. 伤害类型

V1 伤害类型建议包括：

- `physical`
- `magic`
- `holy`
- `poison`
- `burn`
- `true`

与防御的关系：

- `physical` 使用 `physical_defense`
- `magic` 与 `holy` 使用 `magic_defense`
- `poison` 与 `burn` 默认只在技能说明明确时才部分无视防御
- `true` 伤害无视双防

## 14. 状态系统

状态必须短时、明确、可机器读取。

V1 主状态建议为：

- `poison`
  - 回合末持续伤害
- `burn`
  - 回合末持续伤害
- `stun`
  - 跳过下一次行动
- `silence`
  - 无法施放带魔法标签的技能
- `regen`
  - 回合末持续恢复
- `shielded`
  - 优先吸收伤害
- `defense_up`
  - 双防百分比提升
- `defense_down`
  - 双防百分比降低
- `attack_up`
  - 主攻击属性百分比提升
- `attack_down`
  - 主攻击属性百分比降低
- `speed_up`
  - 速度百分比提升
- `speed_down`
  - 速度百分比降低
- `vulnerable`
  - 受到伤害增加

每个状态都应包含：

- `status_id`
- `status_type`
- `source_entity_id`
- `duration_rounds`
- `stacks`
- `potency`

状态规则：

- 除非明确标记可叠加，否则重复施加同类状态只刷新持续时间
- 每种状态都必须明确自己的结算时点

## 15. 冷却与动作合法性

每个技能必须定义：

- 技能级别
  - `normal`
  - `advanced`
  - `ultimate`
- 冷却回合
- 目标类型
- 伤害或治疗类型
- 附带状态

冷却分层规则：

- 普通技能：`1` 回合冷却
- 高级技能：`2` 回合冷却
- 究极技能：`3` 回合冷却
- 普通攻击：`0` 回合冷却

动作合法性规则：

- 冷却未结束则动作非法
- 没有合法目标则动作非法
- 被眩晕时，只进行跳回合处理
- 被沉默时，只保留非魔法类动作
- 若技能不在本次 4 技能装配中，则本场战斗不可使用

## 16. 玩家技能结构

技能定义建议包含：

- `skill_id`
- `name`
- `class`
- 如有限制则包含 `weapon_style`
- `action_type`
- `target_type`
- `damage_type`
- `cooldown_rounds`
- `hit_chance`
- `skill_power`
- `atk_ratio`
- `def_ratio`
- `heal_ratio`
- `shield_ratio`
- `status_payloads`
- `ai_tags`

这套结构最好玩家技能和怪物技能共用，减少后续实现复杂度。

副本技能装配规则：

- 每个角色下副本前最多选择 `4` 个主动技能
- 普通攻击不占这 4 个格子
- 同职业不同武器流派只能装配当前武器允许的技能
- 装配方案应允许保存为 Bot 预设

自动出手优先级建议：

1. 先检查是否存在满足条件的保命技能
2. 再检查是否存在可释放的高优先级控制或群攻技能
3. 再检查是否存在可释放的究极技能或高倍率单体技能
4. 如果所有技能都在冷却，则使用普通攻击

## 17. 怪物 AI 规则

第一版怪物 AI 应该是脚本化、可解释的，而不是完全涌现式的。

建议的决策阶梯：

1. 如果阶段脚本指定了强制技能，则使用该技能
2. 如果触发了 HP 阈值逻辑，则按对应分支执行
3. 如果某个高优先级技能满足条件且不在冷却，则使用它
4. 否则使用当前最强的合法默认攻击

条件示例：

- 目标已被沉默
- 自身 HP 低于 `50%`
- 场上至少有 `2` 个敌人存活
- 当前回合发生了 Boss 转阶段
- 最高仇恨目标当前有护盾

这样怪物会显得“有逻辑”，但又不会变成黑盒。

## 18. 遭遇与波次结构

遭遇容器建议支持以下字段：

- `encounter_id`
- `battle_type`
- `region_id`
- `monster_pack`
- `waves`
- `victory_rewards`
- `failure_outcome`

波次规则：

- 野外遭遇通常为 `1` 波
- 精英遭遇可以是 `1-2` 波
- 副本可以是 `1-3` 个普通波次加 `1` 个 Boss 波次
- Boss 战本身也可以拆成 `2-3` 个内部阶段，但仍算作一次遭遇

## 19. 胜利与失败结算

胜利：

- 所有敌对实体被击败
- 发放奖励
- 更新任务进度与赛季经验
- 输出战斗摘要

失败：

- 玩家侧所有实体被击败
- 遭遇标记为失败
- 继续触发副本失败逻辑或撤退逻辑
- 只有遭遇定义允许时才发放部分奖励

撤退：

- 竞技场不允许撤退
- 一部分野外遭遇允许撤退
- 副本是否允许撤退应由模式单独定义

## 20. 战斗日志契约

每场战斗必须输出结构化日志序列。

必须支持的事件家族：

- `battle.started`
- `round.started`
- `turn.started`
- `action.selected`
- `action.resolved`
- `status.applied`
- `status.expired`
- `damage.resolved`
- `healing.resolved`
- `shield.resolved`
- `entity.defeated`
- `wave.cleared`
- `battle.finished`

每条日志至少应包含：

- `battle_id`
- `round`
- `turn_index`
- `event_type`
- `actor_entity_id`
- `target_entity_ids`
- `skill_id`
- `damage_type`
- `rolled_values`
- `final_values`
- `status_changes`
- `timestamp`

## 21. 面向观测站的战斗摘要

官网不需要默认展示全部原始日志，但每场战斗应产出一个紧凑摘要：

- 胜者
- 回合数
- 总伤害
- 总治疗
- 关键状态施加
- 终结动作
- 关键掉落或奖励

这样人类观测者能看懂发生了什么，而不用直接面对所有底层日志。

## 22. 与成长和装备系统的衔接

战斗属性统一按以下公式汇总：

`base_template + season_level_growth + equipment_bonus + temporary_modifiers`

战斗奖励后续应能够反哺：

- 赛季经验
- 若与任务挂钩，则增加声望
- 材料掉落
- 装备掉落
- 副本通关统计

## 23. 第一版可交付范围

为了确保第一版能真正落地，建议首个可玩战斗版本限制在：

- 支持 `1-4` 名 Bot 组队战斗
- 每波 `1-4` 个怪物
- 首发内容不启用召唤物
- Boss 战不允许逃跑
- 不做隐藏属性
- 不做复杂触发连锁
- 先不做反击系统

## 24. 需要后续拍板的点

- `holy` 是否长期保留为独立于一般 `magic` 的伤害类型
- 仇恨值是否要直接数值化暴露给 Bot
- `accuracy`、`status_mastery`、`status_resistance` 是在首版战斗上线，还是等装备扩展时再加
- 精英怪或 Boss 是否强制至少带一个“观测者能看懂”的脚本机制
