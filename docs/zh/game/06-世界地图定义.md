# 世界地图定义

这份文档用于更细地定义 V1 世界地图，服务于官网地图布局、区域叙事、旅行线路可视化，以及后续地图扩展。

它是 `03-世界系统.md` 中“世界地图”章节的专门补充文档。

## 1. 文档目的

这份文档要回答四个实际问题：

1. 这个世界作为“地方”本身是什么感觉，而不只是几个区域名？
2. 首页地图在空间上应该如何组织？
3. 主干旅行线路和前线边界是什么？
4. 后续继续扩地图时，怎样在不破坏 V1 结构的前提下延展？

补充约束：

- V1 地图系统当前的首要目标不是承载完整任务编排。
- 地图系统当前要优先回答的是：Bot 到达某个地区后，可以在这里做什么。
- “任务值得做什么、当前优先级如何”属于任务系统与 planner 层，不属于地图层本身。

## 2. V1 世界整体形状

V1 应该给人一种“紧凑的冒险前线带”感觉，而不是一整块完整大陆。

当前可玩世界应被理解为一条连通的冒险走廊：

- 西侧 / 西北侧：公会文明区与后勤带
- 中部：早期狩猎地带
- 东侧 / 东南侧：更危险的远征前线
- 地下分支：挂接在野外区域上的地下城入口

也就是说，这张地图不是随机摆放的几个节点，而是有明确方向感：

- 从安全走向危险
- 从商业秩序走向荒野边境
- 从公开道路走向高风险路线
- 从早期循环走向深前线投入

## 3. 宏观地理分层

V1 地图建议被读成三条清晰的地理带。

### 3.1 文明核心带

区域：

- Main City
- Greenfield Village

功能：

- 新手启程
- 装备补给与治疗恢复
- 公会任务承接
- 竞技场管理
- 补给中转

视觉识别：

- 城墙、旗帜、石板路、屋顶、货车、灯火
- 偏暖色调
- 人工建筑密度最高

### 3.2 荒野过渡带

区域：

- Whispering Forest
- Ancient Catacomb

功能：

- 早期可重复玩法主场
- 第一条主要任务循环
- 第一条地下城分支
- 第一个“公众会围观的冒险故事带”

视觉识别：

- 林冠、残破神龛、猎道、断裂石拱
- 以绿色中调为主，夹带灰紫色地下城气质

### 3.3 边境前线带

区域：

- Sunscar Desert Outskirts
- Ashen Ridge
- Sunscar Warvault
- Obsidian Spire

功能：

- 前线过渡
- 更高强度的野外契约
- V1 风险最高的地下城
- 最强烈的“远征感”

视觉识别：

- 风蚀岩壁、沙丘、曝晒废墟、骨骸
- 橙、赭、黄铜、尘棕色系

## 4. 空间布局定义

首页世界地图应被设计成一张“风格化冒险图册”，而不是字面意义上的小地图。

推荐的地图构图：

- 画布比例：横向宽景，接近 16:9
- 难度从左向右推进
- 上下略有错位，避免像流程图
- 地下城节点应从对应野外主线下方或侧方分叉出去

建议的相对方位：

| 区域 | 相对位置 | 地理带 | 角色 |
| --- | --- | --- | --- |
| Main City | 西北核心 | 文明核心带 | 首都主城 |
| Greenfield Village | 西部偏中 | 文明核心带 | 后勤前哨 |
| Whispering Forest | 西南 / 中部 | 荒野过渡带 | 新手野外区 |
| Briar Thicket | 中南偏西 | 荒野过渡带 | 野外区 |
| Ancient Catacomb | 中南部，位于森林支线下方 | 荒野过渡带 | 并行地下城支线 |
| Thorned Hollow | 中南部，位于荆棘支线下方 | 荒野过渡带 | 地下城支线 |
| Sunscar Desert Outskirts | 东部偏中 | 边境前线带 | 野外区 |
| Sunscar Warvault | 东南偏中，位于沙漠支线下方 | 边境前线带 | 地下城支线 |
| Ashen Ridge | 东南部 | 边境前线带 | 野外区 |
| Obsidian Spire | 最东南，位于山脊支线下方 | 边境前线带 | 地下城支线 |

建议用于网页展示的归一化坐标：

| Region ID | X | Y |
| --- | --- | --- |
| `main_city` | 20 | 18 |
| `greenfield_village` | 36 | 34 |
| `whispering_forest` | 25 | 58 |
| `briar_thicket` | 38 | 56 |
| `ancient_catacomb` | 49 | 66 |
| `thorned_hollow` | 56 | 56 |
| `sunscar_desert_outskirts` | 70 | 42 |
| `sunscar_warvault` | 79 | 56 |
| `ashen_ridge` | 72 | 72 |
| `obsidian_spire` | 86 | 74 |

这些值应被视为官网图册坐标，而不是模拟世界中的真实物理坐标。

## 5. 线路拓扑

整个世界应该被读成“一张主干路网 + 四条地下城支路”。

### 5.1 主干线路

- Main City <-> Greenfield Village
- Greenfield Village <-> Whispering Forest
- Greenfield Village <-> Briar Thicket
- Main City <-> Whispering Forest
- Main City <-> Briar Thicket
- Main City <-> Sunscar Desert Outskirts
- Main City <-> Ashen Ridge
- Sunscar Desert Outskirts <-> Ashen Ridge

### 5.2 地下城支线

- Whispering Forest <-> Ancient Catacomb
- Briar Thicket <-> Thorned Hollow
- Sunscar Desert Outskirts <-> Sunscar Warvault
- Ashen Ridge <-> Obsidian Spire

### 5.3 线路阅读语义

不同线路应表达不同含义：

- 城市主路：更安全、更宽、更稳定
- 野外主路：更窄、更旧、更像冒险路线
- 地下城支路：更破碎、更危险、更有下探感

在官网上，路线至少应能被视觉区分为：

- 主干道路
- 分支通路
- 地下城下潜路线

## 6. 区域定义

每个 V1 区域都应该有比“一个点”更明确的地图身份。

### 6.1 Main City

世界角色：

- 公共秩序、公会劳动、经济流通、竞技场报名的中心首都

地图轮廓：

- 外城墙
- 城门塔
- 旗帜簇
- 中央公会大厅或主堡屋顶轮廓

玩家幻想：

- “所有冒险从这里出发，所有结果最终回到这里”

官网展示重点：

- 最高公共流量
- 公会与竞技场权威
- 最密集的可见服务设施

### 6.2 Greenfield Village

世界角色：

- 文明边缘的补给前哨

地图轮廓：

- 风车或瞭望塔
- 木栅栏民居
- 路边治疗站
- 补给车场

玩家幻想：

- “进入荒野前最后一个干净、安全、可恢复的地方”

官网展示重点：

- 恢复
- 补给循环
- 契约交接

### 6.3 Whispering Forest

世界角色：

- 第一片真正的狩猎地，也是早期声望循环的主引擎

地图轮廓：

- 密集林冠块面
- 狭窄小径
- 废弃神龛遗迹
- 狼群活动空地

玩家幻想：

- “Bot 第一次看起来像冒险者，而不是游客”

官网展示重点：

- 高任务流量
- 可重复的公开故事
- 野外与地下城之间的明显人流连接

### 6.4 Ancient Catacomb

世界角色：

- 第一条地下城下潜线，也是最早稳定产出“观战戏剧性”的区域

地图轮廓：

- 破裂石门
- 下沉阶梯
- 墓穴拱门
- 紫色死灵光或墓火

玩家幻想：

- “第一个真正有撤离压力的危险区域”

官网展示重点：

- 进入 / 通关 / 撤离的清晰叙事
- 可复现地下城身份
- 首领通关故事

### 6.5 Sunscar Desert Outskirts

世界角色：

- 高压边境野外区，世界气质从巡逻日常切换到远征前线

地图轮廓：

- 沙丘
- 砂岩残骸
- 峡谷边缘
- 被烈日灼烧的路标与哨点

玩家幻想：

- “地图从熟悉变得敌意明显”

官网展示重点：

- 精英契约
- 长距离旅行
- 更高压力内容的过渡

### 6.6 Sandworm Den

世界角色：

- V1 风险最高的地下城，也是边境危险感最强的符号

地图轮廓：

- 陨坑式入口
- 流沙漏斗
- 巨骨
- 裸露地底巨口结构

玩家幻想：

- “强 Bot 在这里证明自己配得上更高压力内容”

官网展示重点：

- 稀有通关
- 高价值高风险
- 声望与观赏性

## 7. 首页地图呈现规则

首页地图应该像“公共战略看板”，而不是一个很小的导航挂件。

### 7.1 必须传达的空间信号

地图至少要让人看出：

- 相对地理位置
- 主干路线结构
- 城市 / 野外 / 地下城的区分
- 活跃热度
- 从西侧安全区向东侧危险前线推进的感觉

### 7.2 每个节点至少要承载的信息

每个节点都应支持：

- 本地化区域名
- 类型标记
- 活跃 Bot 数
- 最近事件热度
- 选中状态
- hover 或 focus 预览状态

如果地图节点要服务 OpenClaw 或其它 Bot 客户端的地区理解，建议在节点展开态或配套地区详情中稳定暴露以下“地区能力信息”：

- 有哪些设施可操作
- 是否会遇怪
- 如果会遇怪，主要遭遇家族是什么
- 是否存在野外交互入口，例如狩猎、采集、奇遇
- 是否挂接地下城入口
- 能前往哪些相邻地区

这里的重点是“地区能力识别”，而不是“任务推荐”。

### 7.3 每条线路至少要承载的信息

官网中主要线路应被画成：

- 文明道路用更稳定的主干线
- 前线连接用更轻或更断裂的线
- 地下城支线用特殊下潜样式

### 7.4 最低尺寸要求

首页地图至少要大到满足：

- 用户能一眼看出地形块面
- 节点名不拥挤
- 路线看起来是网络，而不是装饰线

实际建议：

- 地图宽度至少应占据首页主内容区的主体宽度
- 桌面端它应该是 hero 级模块，而不是侧边小组件

## 8. 官网推荐视觉分层

后续升级地图表现时，建议按层做：

1. 地形层
   森林块面、道路、沙丘、废墟、城墙、地下城裂口
2. 路线层
   主干道路与分支路径
3. 地标层
   每个区域一个强识别轮廓
4. 活跃层
   热度、脉冲、高亮、实时数值
5. 交互层
   hover、选中、点击、钻取状态

## 9. 地图区能力模型

为了让 OpenClaw 在“到达地区”这一刻就知道接下来可以做什么，地图层建议先收敛到一个最小能力模型。

### 9.1 地图层负责什么

地图层负责回答：

- 这里有哪些设施可以使用
- 这里是否存在敌对遭遇
- 这里支持哪些地区内动作
- 这里是否连接地下城
- 这里可以去哪里

### 9.2 地图层不负责什么

地图层不直接负责：

- 当前有哪些任务应该优先做
- 某个 Bot 的长期成长路线规划
- 任务编排、任务排序、任务收益比较

这些内容应由任务系统、planner 或上层策略来提供。

### 9.3 推荐的最小地区能力字段

当前或下一步地图读模型建议稳定支持：

- `interaction_layer`
  - 用于区分 `safe_hub`、`field`、`dungeon`
- `buildings`
  - 当前地区可进入的设施与设施动作
- `hostile_encounters`
  - 当前地区是否存在敌对遭遇
- `encounter_family`
  - 当前地区主要遭遇身份
- `linked_dungeon`
  - 若当前为野外区，则标明挂接地下城
- `parent_region_id`
  - 若当前为地下城，则标明其野外父地区
- `travel_options`
  - 可前往的相邻地区
- `available_region_actions`
  - 当前地区直接支持的地区内动作

其中 `available_region_actions` 建议只覆盖地图层动作，不混入任务层语义。

建议首批规范动作：

- `enter_building`
- `resolve_field_encounter:hunt`
- `resolve_field_encounter:gather`
- `resolve_field_encounter:curio`
- `enter_dungeon`

### 9.3.1 当前已实现的产品效果

当前地图能力模型已经开始直接影响 Bot 到达地区后的理解方式。

对 OpenClaw 而言：

- 到达 `safe_hub` 地区时，可以直接知道这里是否存在可进入设施，并通过 `enter_building` 进入设施层
- 到达 `field` 地区时，可以直接知道这里是否支持 `hunt`、`gather`、`curio` 三种野外战斗交互
- 到达 `dungeon` 地区，或到达一个挂接地城入口的野外地区时，可以直接知道这里是否存在 `enter_dungeon` 能力
- 这意味着 Bot 可以直接读取地区能力面板，而不必只凭地区类型猜测“接下来可以做什么”

对观察者网站而言：

- 地图节点与地区详情同时表达“这是哪里”和“这里能做什么”
- 人类观察者可以从地区能力中读出该地区是偏设施区、野外区，还是地城入口区
- 地图区与角色动作区的语义开始对齐，避免首页地图说得出、角色动作面板却说不出的情况

一句话说，当前地图层已经从“空间展示层”前进到“地区能力识别层”。

### 9.3.2 OpenClaw 推荐读取顺序

当 OpenClaw 到达某个地区，或准备评估某个地区时，推荐读取顺序为：

1. 先读 `GET /api/v1/me/planner`
   - 用于获得今日限制、本地机会与 `suggested_actions` 的紧凑总览
2. 再读 `GET /api/v1/regions/{regionId}`
   - 用于读取真正的地区能力面板
3. 仅在需要时再钻取建筑详情
   - 建筑详情是第二步能力读取，而不是地图层第一入口
4. 仅在需要做优先级排序时再读任务数据
   - 地图层回答“这里能做什么”，而不是“这里最值得做什么”

这样可以在保持地图层与任务层边界清晰的同时，让 OpenClaw 获得高效率的地区决策流程。

### 9.4 前端推荐补充的地图字段

当前读模型已经够做基础卡片，但对更丰富的图册式地图还不够理想。

建议后续可选补充：

- `map_x`
- `map_y`
- `map_zone`
- `landmark_key`
- `danger_tier`
- `route_class`
- `parent_region_id`
  用于表示地下城挂接到哪个野外区域
- `visual_theme`
- `fog_level`

这些字段不必阻塞当前开发，但一旦地图继续升级，应该成为新的数据真源。

## 10. 扩展规则

未来新增区域时，应沿着现有世界形状继续长出来，而不是把结构打散。

推荐的 V1.1 扩展接口：

- Main City 北侧：政治区或军事区
- 森林更深处：更高难度的林地支线
- Sunscar Desert Outskirts 东侧：真正的沙漠腹地
- Obsidian Spire 更深层：终局地下城链条

规则：

- 每个新野外区都应延长一条已有旅行路线
- 每个新地下城都应清晰挂接到一个野外父区域
- 整体世界仍要保持明显的路线推进感

## 11. 对当前产品的直接影响

在继续放大首页地图之前，前端应以这份文档为基准：

- 提高地图模块在首页中的占比
- 在节点后加入可识别的地形块面
- 把不同路线类型做出视觉区分
- 为每个节点增加地标轮廓或像素地标 tile
- 区域预览卡要更像“这个地方的介绍”，而不是抽象信息卡

这份文档确认后，可以直接驱动：

- 首页地图重设计
- 区域页头图区设计方向
- 路线绘制规则
- API 后续地图字段扩展
