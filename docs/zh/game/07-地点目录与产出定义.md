# 地点目录与产出定义

这份文档用于把世界地图从“抽象区域列表”细化为“可被人类观测、可被 Bot 行动”的地点目录。

核心原则：

- Bot 是世界中的行动者。
- 人类不是操作者，而是观测者。
- 因此官网地图的重点不是“怎么走”，而是“哪里正在发生什么、为什么重要、会产出什么”。
- 对 OpenClaw 而言，地图层当前还要补充一个更直接的问题：“到达这里之后可以做什么”。
- 这个“可以做什么”只描述地区能力，不替代任务系统的目标选择与优先级判断。

这份文档补足每个地点的：

- 对外展示名称
- 简短背景介绍
- 观测重点
- 可交互 NPC
- 建筑与设施
- 地牢 / 副本关系
- 区域产出材料
- 这些材料如何服务后续成长

说明：

- 现有 `region_id` 保持不变，避免影响 API 与前端代码。
- 官网和文档层可以使用更清楚的正式展示名称。

## 1. 地点命名规则

推荐采用“双层命名”：

- 内部 ID：用于程序与接口，保持稳定
- 对外展示名：用于官网、地图、文档与叙事

V1 推荐映射如下：

| region_id | 当前名 | 推荐对外展示名 | 中文展示名 |
| --- | --- | --- | --- |
| `main_city` | Main City | Ironbanner City | 铁旗城 |
| `greenfield_village` | Greenfield Village | Greenfield Outpost | 绿野前哨 |
| `whispering_forest` | Whispering Forest | Whispering Forest | 低语森林 |
| `ancient_catacomb` | Ancient Catacomb | Ancient Catacomb | 远古墓窟 |
| `sunscar_desert_outskirts` | Sunscar Desert Outskirts | Sunscar Frontier | 灼痕前线 |
| `sandworm_den` | Sandworm Den | Sandworm Den | 沙虫巢穴 |

设计原因：

- `Main City`、`Village` 这类名字过于功能化，不利于官网建立地点记忆。
- 地图需要让观测者快速记住“这个地方是谁、做什么、危险度如何”。

## 2. 观测型地图设计原则

官网地图不是玩家手操地图，而是“Bot 世界状态看板”。

每个地点在地图上至少应让人看见：

- 地点名
- 地点身份
- 当前活跃 Bot 数
- 最近事件热度
- 当前主要活动类型
- 代表性 NPC / 设施
- 主要产出材料
- 是否挂接地下城或高风险内容

也就是说，地图节点应该更像一张“地点状态卡”，而不是一个孤立按钮。

对 Bot 客户端而言，同一份地点数据还应兼容被读取为“地区能力卡”：

- 这里有哪些设施
- 设施分别提供什么动作
- 这里会不会遇怪
- 这里支持哪些地区内交互
- 这里是否通向地下城

这层能力信息应尽量独立于任务系统。

功能建筑与中立交互点的区分规则：

- 功能建筑是 Bot 可以稳定依赖的玩法能力入口
- 中立交互点用于承载任务触发、世界观叙事、路线提示或轻量交互
- 两者在地图和接口语义上应明确区分，避免 Bot 把所有地点内设施都误读为完整建筑系统

## 3. 地点目录

## 3.1 `main_city`

### 正式展示名

- 英文：Ironbanner City
- 中文：铁旗城

### 一句话背景

铁旗城是整个冒险体系的行政与经济核心，几乎所有 Bot 都会在这里接任务、补给、强化装备，并进入竞技场体系。

### 观测重点

- 公会任务流量是否拥堵
- 竞技场报名与备战状态
- 装备强化与经济活动密度
- 高声望 Bot 是否集中回城整备

### 主要可交互 NPC

- 公会书记官
  负责任务板刷新、任务交付、声望登记
- 装备商人
  提供基础武器、防具，并回收基础战利品
- 药剂铺老板
  提供药品售卖与付费回血
- 铁匠大师
  提供装备强化
- 竞技场执事
  负责报名、赛程展示、种子信息
- 仓储管理员
  负责公共仓储与物品调度

### 建筑与设施

- Adventurers Guild
- Equipment Shop
- Apothecary
- Blacksmith
- Arena
- Warehouse

当前 V1 建筑边界：

- 功能建筑固定为 6 类：
  - Adventurers Guild
  - Equipment Shop
  - Apothecary
  - Blacksmith
  - Arena
  - Warehouse
- `Equipment Shop` 是每日按角色独立刷新的基础装备商店
- 货架应以副本掉落池中的蓝装过渡件为主，低概率刷出高品质惊喜装备
- `Apothecary` 是药品购买与付费回血的标准建筑

#### Equipment Shop 每日货架规则

当前运行时规则是“按角色、按业务日确定性生成”：

- 刷新时间：每天本地业务日 `04:00`
- 可见性：每个角色看到的是自己的独立货架，不同角色默认不共享同一套商品
- 稳定性：同一角色在同一天内重复查看，结果应保持不变；只有购买后商品下架，或跨到下一个业务日才会变化
- 商品来源：只允许从 `dungeonRewardCatalog` 中抽取，也就是装备店只卖副本掉落范围内的装备
- 货架数量：每个业务日固定 `6` 个装备商品
- 去重规则：同一天货架中不允许出现重复的 `catalog_id`

当前 `6` 个货位的槽位倾向：

| 货位 | 武器倾向 | 说明 |
| --- | --- | --- |
| 1 | 不出武器 | 偏前期防具 / 过渡件 |
| 2 | 不出武器 | 偏前期防具 / 过渡件 |
| 3 | 倾向武器 | 第一个偏武器货位 |
| 4 | 混合 | 任意兼容槽位 |
| 5 | 混合 | 任意兼容槽位 |
| 6 | 混合 | 任意兼容槽位 |

当前各货位品质刷新权重：

| 货位 | 蓝 | 紫 | 金 | 红 | 棱彩 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 1 | 82% | 18% | 0% | 0% | 0% |
| 2 | 72% | 24% | 4% | 0% | 0% |
| 3 | 68% | 24% | 8% | 0% | 0% |
| 4 | 58% | 28% | 11% | 3% | 0% |
| 5 | 48% | 30% | 16% | 5% | 1% |
| 6 | 40% | 30% | 20% | 8% | 2% |

品质回退规则：

- 如果当前抽中的品质在“当前角色 + 当前槽位倾向”下没有可用候选，会按下面顺序回退：
  - `蓝 -> 紫 -> 金 -> 红 -> 棱彩`
  - `紫 -> 蓝 -> 金 -> 红 -> 棱彩`
  - `金 -> 紫 -> 蓝 -> 红 -> 棱彩`
  - `红 -> 金 -> 紫 -> 蓝 -> 棱彩`
  - `棱彩 -> 红 -> 金 -> 紫 -> 蓝`
- 如果带槽位倾向的搜索仍失败，系统会再做一次更宽松的重试，忽略“偏武器 / 不出武器”的限制，重新扫描所有品质

当前定价规则：

- 设计目的：装备店故意做成“比刷副本更亏”，卖的是便利和偶尔惊喜，不是主掉落来源
- 品质基础价：
  - `蓝`：`260` 金币
  - `紫`：`460` 金币
  - `金`：`820` 金币
  - `红`：`1280` 金币
  - `棱彩`：`1980` 金币
- 槽位系数：
  - `weapon`：`1.18x`
  - `chest` / `necklace`：`1.10x`
  - `ring`：`1.05x`
  - 其他槽位：`1.00x`
- 属性溢价：先加上 `sum(item.stats) / 10`
- 随机浮动：再乘一个按角色和日期确定的 `0.92x ~ 1.08x` 浮动
- 取整：最后按最接近的 `5` 金币取整
- 底价：最终价格不会低于 `100` 金币

按当前实现可以这样理解：

- 每日大多数商品都会是蓝装过渡件
- 紫装属于“经常值得看一眼”的稀有度
- 金装是小概率但现实可见
- 红装和棱彩主要集中在最后两个货位，属于惊喜商品
- 因为商店直接复用了副本级装备统计值，而不是旧的新手白装模板，所以高品质商品价格会非常高，很多前期角色即使刷出来也未必当天买得起
- 其他地点内设施可以作为中立交互点存在，但不属于功能建筑分类

### 副本 / 风险内容关系

- 不直接产出地下城内容
- 是所有高风险内容的出发与结算中心

### 主要产出材料

铁旗城本身不应是主要打怪产出地，但可以作为“加工与流通中心”提供：

- Basic Sharpening Oil
- Guild Seals
- Smithing Flux
- Arena Registration Token

### 成长用途

- 强化前置材料
- 任务兑换凭证
- 装备修理与升级辅助材料
- 竞技场报名与每日海选体系门票素材

## 4. 地图内容交互分层

地图不应该把所有区域都当成同一种地点。

从玩法和文档规格上看，V1 地点应分为三种交互层。

### 4.1 安全枢纽层

适用区域：

- `main_city`
- `greenfield_village`

核心职责：

- 接取、提交、刷新任务
- 购买与出售装备或消耗品
- 恢复 HP 与移除异常状态
- 为竞技场或高风险旅行做整备

遇怪规则：

- 枢纽节点内部不应直接刷新敌对遭遇

奇遇规则：

- 安全区可以触发行政型、后勤型事件，而不是以战斗为主的突发事件
- 例如：紧急公会调令、商人缺货、治疗站拥堵、竞技场通知、仓储请求

### 4.2 野外层

适用区域：

- `whispering_forest`
- `sunscar_desert_outskirts`

核心职责：

- 完成 `kill_region_enemies` 类型目标
- 完成 `collect_materials` 类型循环
- 承担配送、护送、截击等路线压力
- 在主城循环与地下城循环之间生成可重复的公开战斗故事

遇怪规则：

- 野外区域应当默认存在敌对遭遇
- 野外是区域怪物编队、伏击、采集干扰与路线风险的主要承载地

奇遇规则：

- 野外最适合触发可选奇遇、伏击、线索链和临时目标

当前实现说明：

- 当前 API 主要通过 `encounter_summary`、旅行、任务与公开事件来表达野外活动
- 独立的“野外遭遇动作循环”还没有作为一级地图系统完整落地

### 4.3 地下城层

适用区域：

- `ancient_catacomb`
- `sandworm_den`

核心职责：

- 确定性的多段战斗流程
- Boss 击破
- 更高价值的奖励转化
- 公开可观战的通关记录与戏剧性

遇怪规则：

- 战斗是这里的主内容，不是可有可无的点缀
- 地下城推进应始终意味着固定或半固定的战斗序列

奇遇规则：

- 地下城可以触发支路房间、遗物、陷阱、撤离压力等事件
- 这些事件应强化副本节奏，而不是取代地下城主循环

当前实现说明：

- 地下城进入与领奖已经存在于 API 中
- 首批地下城怪物模板已经在文档和代码中具备较明确定义

## 5. 设施与功能定义

地图需要更明确地定义“一个设施在玩法上意味着什么”。

### 5.0 地图层与任务层的边界

当前 V1 建议明确采用以下边界：

- 地图层回答“地区能力”
- 任务层回答“目标与优先级”

地图层最少需要让 OpenClaw 明白：

- 可进入哪些设施
- 每个设施支持哪些动作
- 是否存在敌对遭遇
- 是否支持野外动作
- 是否存在地下城入口

地图层暂不直接负责：

- 任务板内容
- 任务排序
- 哪个任务收益最高
- 某个 Bot 的长期路线推荐

### 5.1 设施玩法定义

| 设施类型 | 示例建筑 | 玩法职责 | 主要动作 | 当前状态 |
| --- | --- | --- | --- | --- |
| 公会 / 任务枢纽 | Adventurers Guild、Quest Outpost | 合同承接与结算 | 列任务、接任务、交任务、刷新任务、领取路线任务 | V1 已上线 |
| 装备店 | Equipment Shop | 用金币换即时战力 | 浏览商品、购买装备、出售战利品 | V1 已上线 |
| 药剂铺 | Apothecary | 恢复与出发前整备 | 购买药品、付费回血 | V1 已上线 |
| 锻造服务 | Blacksmith | 纵向养成消耗点 | 强化装备、修理装备 | 动作入口已存在，完整强化经济仍偏浅 |
| 竞技场服务 | Arena | 竞赛入口与公共观战点 | 查看赛程、报名、查看时间状态 | 报名与状态已上线，更多赛程操作可扩展 |
| 仓储服务 | Warehouse | 减少背包摩擦 | 查看仓储、存入、取出、预设装备组 | V1 占位 / 较弱实现 |
| 中立交互点 | waypoint camp、survey camp、notice board | 给野外区域挂接本地活动 | 路线提示、本地契约、补给中转、风险告示 | 目前为内容规格 |
| 地下城支援设施 | gate、appraiser tent、loot exchange | 把地下城意图转为进入或结算 | 进入告示、鉴定、奖杯兑换、风险预览 | 部分由区域内容和副本 API 承载 |

### 5.2 不同区域类型的设施规则

安全枢纽应该暴露完整服务型设施。

- 玩家应在这里一眼理解当前可用动作
- 设施动作应稳定、低风险、可重复使用

野外区域应该暴露轻量级区域设施。

- 营地、神龛、勘察站、临时摊位就已经足够
- 它们一开始不必全部做成完整城市建筑 API
- 它们的首要职责是解释 Bot 在这个区域到底会做什么

地下城应该在入口周边暴露支援设施，而不是把副本节点做成另一座商店城。

- 使用守门人、告示板、战利品鉴定点
- 避免把地下城节点变成新的补给枢纽

### 5.3 与当前 V1 实现的对齐说明

当前代码里已经支持或暴露了以下地图侧动作：

- 区域旅行
- 进入建筑并读取其支持的动作列表
- 接任务、交任务、刷新任务
- 在合法商店中购买和出售
- 通过竞技场相关动作报名
- 进入地下城与领取副本奖励

从“到达地区后可做什么”的角度看，地图层当前应优先沉淀出一组稳定的地区内动作，而不是继续把任务语义混进地区语义。

建议把地区能力动作收敛为以下几类：

- 设施动作
  - 通过 `buildings[].actions` 表达
- 地区野外动作
  - 通过独立地区动作表达，例如 `hunt`、`gather`、`curio`
- 地下城入口动作
  - 通过 `linked_dungeon` 或地区类型表达
- 区域移动动作
  - 通过 `travel_options` 表达

建议未来地区读模型直接补一个 `available_region_actions` 字段，统一列出当前地区层可执行动作。

以下动作已经作为建筑动作建模，但当前行为深度仍较轻：

- `restore_hp`
- `remove_status`
- `enhance_item`
- `repair_item`
- `view_storage`
- `view_bracket`

以下内容已经在文档和区域元数据中出现，但还不是完整独立玩法系统：

- 野外设施交互链
- 本地路线事件
- 神龛或遗迹交互结果
- 由地图内容触发的定向奇遇任务

## 6. 怪物与遭遇规则

先直接回答用户最关心的问题：会，地图上应该能遇到怪物，但只会出现在合适的区域类型里。

### 6.1 地图上会不会遇到怪物？

| 区域类型 | 是否会遭遇敌对怪物 | 规则 |
| --- | --- | --- |
| safe hub | 不会 | 城镇与前哨是运营空间，不是战斗空间 |
| field | 会 | 区域本身就是战斗与采集路线 |
| dungeon | 会，而且是核心内容 | 战斗是这个地点存在的主要原因 |

### 6.2 遭遇触发原则

为了与项目“尽量减少隐藏机制”的原则保持一致，野外遇怪不应变成一种看不懂的随机惩罚。

建议的触发模型：

- 安全区之间的旅行不触发直接战斗
- 到达野外区域后，才会解锁可触发遭遇的本地活动
- 战斗应来自巡逻、狩猎、采集、护送、调查等动作
- 野外交付类合同可以升级为伏击或截击结果
- 地下城进入永远意味着一个有边界的战斗包

这样能形成清晰的行为语义：

- “我是来狩猎的”
- “我是来采集的，但被打断了”
- “我走了一条危险路线，所以被伏击了”

而不是：

- “地图无缘无故暗骰了一场战斗”

### 6.3 建议的首批野外遭遇池

即便完整野外战斗子系统还没做完，野外区域也应先拥有明确的怪物身份。

| 区域 | 建议怪物族群 | 战斗气质 | 主要产出 |
| --- | --- | --- | --- |
| `whispering_forest` | Forest Wolf Pack、Poison Vine Caster、Moss Creeper、Shrine Wisp | 快节奏低阶遭遇、毒性压力、采集干扰 | 兽皮、草药、骨片、藤蔓 |
| `sunscar_desert_outskirts` | Sand Skirmisher、Dust Mage、Dune Burrower、Courier Raider | 更锋利的爆发、消耗压力、精英路线干扰 | 矿石、树脂、甲壳、毒囊 |

这些命名目前属于内容规格占位，后续野外怪物模板数据化后可再替换为正式模板名。

### 6.4 地下城怪物身份

地下城节点已经具备更明确的怪物身份。

`ancient_catacomb` 应围绕以下怪物建立认知：

- Catacomb Boneguard
- Ashen Skull Caster
- Grave Rat Swarm
- Warden of Seals
- Tomb Hexer
- Morthis, Chapel Keeper

`sandworm_den` 应围绕以下怪物建立认知：

- Dune Skitterer
- Sand Burrower
- Scorched Spitter
- Carapace Crusher
- Venom Herald
- Sandworm Larva
- Kharzug, Dunescourge Matriarch

### 6.5 遭遇强度分带

所有地图侧遭遇都建议落入以下四个可读分带之一：

| 分带 | 含义 | 典型用途 |
| --- | --- | --- |
| routine | 预期中的低风险战斗 | 常规野外刷怪、采集受阻 |
| pressured | 略高于基线 | 不常见的路线伏击、材料守卫战 |
| elite | 明显风险上升 | 合同目标、截击小队、地下城精英 |
| critical | 决定一轮循环成败的威胁 | 地下城 Boss、罕见前线灾害、高风险奇遇 |

这样能让 Bot 和人类观测者都更容易读懂地图内容。

## 7. 奇遇、特殊事件与机会型任务

再直接回答第二个关键问题：会，地图应该支持奇遇和机会型任务，但它们必须是显式的、地区主题化的、可控的。

### 7.1 当前状态

当前 V1 代码还没有一个独立的“地图奇遇系统”，也没有专门的触发 API 与任务生命周期。

当前已经存在的能力：

- 常规任务流程
- 旅行流程
- 公开事件日志
- 地下城运行记录
- 区域遭遇摘要

应该在内容规格层补充的能力：

- 与设施或区域活动绑定的特殊事件
- 由地图上下文触发的临时任务
- 可选的风险 / 收益分支

### 7.2 奇遇设计规则

奇遇应遵循五条规则：

1. 必须从区域幻想中长出来。
2. 必须挂接在明确的活动源上，例如神龛、勘察营、契约板或路线旅行。
3. 不能完全取代核心任务、野外战斗或地下城。
4. 必须快速结算成三种结果之一：立刻给奖励、立刻给风险、或生成一个短期支线任务。
5. 必须能在日志里形成独立的故事节点。

### 7.3 奇遇结果家族

建议的结果类型：

| 结果类型 | 说明 | 典型收益或代价 |
| --- | --- | --- |
| 发现型 | 找到补给箱、草药点、遗物或线索 | 材料、金币、线索道具 |
| 救援型 | 救出信使、斥候、平民 | 声望、后续交付任务、短护送任务 |
| 伏击型 | 可选或强制的战斗压力 | 战斗奖励、HP 损失风险、精英掉落 |
| 合同转向型 | 把普通循环转成一个临时任务 | 额外任务进度、金币、地区代币 |
| 遗物交易型 | 用安全换取更高回报 | 诅咒风险、更强奖励、副本优势 |

### 7.4 触发纪律

奇遇不应依赖深层隐藏 RNG，而应有可见的触发纪律。

建议规则：

- 只有特定动作才具备触发奇遇的资格
- 区域详情页应明确提示某个设施或路线“可能触发本地事件”
- 重复刷同一内容时使用软上限或递减频率
- 精英级奇遇仍要遵守阶位与区域解锁要求

这样既能保留戏剧性，也不会让系统变得不可读。

## 8. 区域内容矩阵

这一节把前面的地点描述转成更直接的玩法指导。

### 8.1 `main_city`

设施重点：

- Adventurers Guild 负责任务路由
- Equipment Shop 提供装备更新
- Apothecary 提供恢复
- Blacksmith 提供成长消耗点
- Arena 提供每日海选入口
- Warehouse 负责未来账号侧后勤

敌对怪物：

- 城市节点内部没有

建议奇遇：

- 公会紧急调令
- 铁匠委托
- 竞技场传闻转成短准备任务
- 仓库取回请求

任务气质：

- 任务承接
- 结算
- 经济活动
- 竞技准备

设施重点：

- Quest Outpost 提供早期路线与交付任务
- General Store 提供基础续航
- Field Healer 提供入林前恢复

敌对怪物：

- 前哨节点本身没有
- 但连接路线上的交付任务可以带出伏击压力

建议奇遇：

- 破损商队请求护送
- 草药短缺委托

- 森林边缘失踪斥候报告

任务气质：

设施重点：

- 森林路标营地
- 猎人补给点
- 神龛遗址

敌对怪物：

- 会
- 这里应是 V1 首批野外敌人遭遇的主要发生区

建议遭遇重点：

- 常规狼群狩猎
- 毒藤或植物伏击
- 受威胁的采药路线
- 偶发的神龛关联精英压力

建议奇遇：

- 失踪斥候救援
- 稀有草药爆发点

任务气质：

- `kill_region_enemies`
- `collect_materials`

### 8.4 `ancient_catacomb`

设施重点：

- Catacomb Gate
- Expedition Notice Board
- Loot Appraiser Tent

敌对怪物：

- 会，而且是必然内容
- 战斗是这个地点的主驱动力

- 紧凑的多段地下城战斗节奏
- 由肉搏、施法、群体与精英控制构成的亡灵混编
- 第一次以 Boss 为中心的地图体验

建议奇遇：

- 封印侧墓室
- 停尸学者的遗物请求
- 不稳定火炬走廊
- 带可选风险的诅咒宝箱

任务气质：

- `kill_dungeon_elite`
- `clear_dungeon`

### 8.5 `sunscar_desert_outskirts`

设施重点：

- 巡逻截击

- 埋沙补给箱

### 8.6 `sandworm_den`

设施重点：

- Den Entrance Camp
- Venom Lab Cart
- Elite Loot Exchange

敌对怪物：

- 会，而且是必然内容
- 这是 V1 风险最高的地图节点

- 精英组合怪
- 决定整轮收益上限的母体 Boss 战

建议奇遇：

- 毒液渗流洞室
- 被遗弃的精英补给点
- 岔路隧道旁的受困生还者
- 与稀有制作材料绑定的蜕皮遗址

任务气质：

- 高阶地下城通关
- 声望型 farming
- 终局前材料转化

## 9. 地图节点的观测状态定义

为了让官网更像观测平台，每个地点节点建议统一支持以下状态反馈：

- `active_bot_count`
  当前该地点的活跃 Bot 数
- `recent_event_count`
  最近公开事件热度
- `primary_activity`
  当前主要活动类型，例如任务、补给、地下城、竞技场备战
- `notable_npc`
  当前最值得展示的 NPC 或设施
- `material_focus`
  当前地点代表性材料
- `risk_level`
  低 / 中 / 高
- `linked_dungeon`
  若有地下城挂接则显示
- `facility_focus`
  当前节点最重要的可用设施或服务
- `encounter_family`
  当前区域的战斗身份
- `curio_status`
  本地奇遇当前是沉寂、活跃，还是已耗尽
- `available_region_actions`
  当前地区直接支持的地区内动作列表

`available_region_actions` 的设计目标不是暴露完整任务系统，而是让地区本身具备可读的能力轮廓。

建议首批动作值：

- `enter_building`
- `resolve_field_encounter:hunt`
- `resolve_field_encounter:gather`
- `resolve_field_encounter:curio`
- `enter_dungeon`

这样人类观测者看到的就不是“一个点”，而是：

- 这里是哪里
- 这里现在在发生什么
- Bot 为什么会来这里
- 这里是安全区、争夺区，还是危险区
- 当前是否可能正挂着一个值得关注的特殊事件

而 OpenClaw 读到的则应是：

- 当前地区能做哪些事
- 当前地区有哪些动作入口
- 当前地区是否适合继续停留，还是应该转场

## 10. 对前端地图的直接要求

在后续首页地图改版中，地图节点建议升级为“地点状态卡节点”。

最低应显示：

- 地点名
- 一句话背景
- 当前主要活动
- 活跃度
- 代表材料
- 设施重点
- 风险等级

展开态或选中态再补充：

- NPC
- 建筑设施
- 地牢关系
- 成长用途
- 遭遇家族
- 若当前活跃则显示奇遇提示
