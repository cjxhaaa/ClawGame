# 自动战斗推演流程说明

## 1. 适用范围

本文用于按步骤说明自动战斗系统的运行流程。

它是以下文档的流程层补充：

- `10-战斗系统框架.md`（规则与契约）
- `15-战斗消耗品与药水系统.md`（续航与药水规则）

## 2. 运行时输入

每场战斗启动时的输入：

- 战斗元信息（`battle_type`、`encounter_id`、`region_id`、可选 `dungeon_id`）
- 玩家队伍快照（属性、装配技能、状态、背包相关标记）
- 敌方波次定义（怪物模板、AI 标签、波次脚本）
- 系统配置（回合上限、动作合法性规则、随机种子策略）

## 3. 初始化阶段

初始化顺序：

1. 创建战斗上下文并生成 `battle_id`
2. 按模板和运行时修正实例化实体
3. 设置 `current_hp`、`shield_value`、冷却和初始状态
4. 锁定技能装配
5. 构建波次状态与当前参战列表
6. 输出 `battle.started`

初始化产物：

- 归一化后的战斗内存状态
- 首回合行动顺序预览（可选）
- 首批日志事件

## 4. 主循环

主循环持续运行，直到命中终止条件。

终止条件：

- 玩家侧胜利
- 敌方胜利
- 达到回合上限且敌方仍存活（判负）
- 命中配置的硬失败条件

高层循环：

1. 回合开始
2. 计算行动顺序
3. 单位行动循环
4. 回合结束
5. 终止检测

## 5. 回合开始阶段

回合开始动作：

1. 回合数加一
2. 结算明确声明为“回合开始触发”的周期效果（例如 `owner_turn_start`）
3. 移除或降级在回合开始到期的状态
4. 输出 `round.started`

如果开始效果后任一方已全灭，直接进入终止检测。

## 6. 行动顺序计算

顺序规则：

- 先按 `speed` 降序，再按 `entity_id` 升序

补充说明：

- 每回合重算顺序
- 速度增减益影响下一次重算
- 无法行动的单位仍保留在序列中并消耗回合

## 7. 单位行动管线

对当前顺序中的每个存活单位依次执行：

1. 输出 `turn.started`
2. 判断硬阻断（`stun`、死亡、强制跳过）
3. 生成合法动作集合
4. 按自动策略选动作（玩家策略或怪物脚本）
5. 输出 `action.selected`
6. 执行动作结算
7. 输出一组结算事件

### 7.1 合法动作集合构建

候选动作：

- 普通攻击
- 已装配主动技能
- 满足规则时的药水动作

过滤非法动作：

- 冷却中
- 无合法目标
- 沉默下不可用技能
- 本次副本限制（例如药水次数上限）

### 7.2 自动选招规则

玩家侧优先级：

1. 紧急保命
2. 团队恢复/护盾
3. 控制或群体收益
4. 高价值终结输出
5. 普攻兜底

怪物侧优先级：

1. 脚本强制动作
2. 阶段阈值分支
3. 高优先级条件技能
4. 默认攻击兜底

## 8. 动作结算管线

每个选中动作按以下顺序结算：

1. 解析目标
2. 如有命中判定则先掷命中
3. 计算原始数值（伤害/治疗/护盾）
4. 应用乘区与伤害类型交互
5. 先结算护盾，再结算 HP 变化
6. 施加状态载荷
7. 更新冷却与资源
8. 输出结算日志（`damage.resolved`、`healing.resolved`、`shield.resolved`、`status.applied` 等）

若目标 HP 归零：

- 标记 `alive = false`
- 输出 `entity.defeated`

## 9. 回合结束阶段

回合结束动作：

1. 结算明确声明为“回合结束触发”的周期效果
2. 递减持续效果的剩余回合
3. 清理到期状态并输出 `status.expired`
4. 输出回合总结块（若开启）

之后执行终止检测。

持续效果解释：

- 周期触发型效果与状态型效果共用同一套“有效窗口”基线：施法者本次出手到该施法者下一次出手之间，算作 `1` 回合
- 周期触发型效果不会持续不断地跳字，只会在自己声明的触发阶段结算
- `silence`、`defense_up` 这类状态型效果在整个有效窗口内持续生效，但不会自行产出周期性伤害或治疗事件

## 10. 终止结算

### 10.1 胜利

当敌方全灭：

1. 标记战斗结果为胜利
2. 结算奖励与进度
3. 输出 `battle.finished`
4. 生成观测摘要

### 10.2 失败

当玩家侧全灭，或触发回合上限失败：

1. 标记战斗结果为失败
2. 执行失败后果配置
3. 输出 `battle.finished`
4. 生成观测摘要

## 11. 副本房间串联

对多房间 `dungeon_wave`：

1. 在房间间持久化队伍战斗状态
2. 按副本规则继承 HP/护盾/状态
3. 装载下一房间波次并重新进入初始化

HP 继承和药水行为统一遵循：

- `15-战斗消耗品与药水系统.md`

## 12. 日志检查清单

一次常规单位行动的最小事件序列：

1. `turn.started`
2. `action.selected`
3. 以下至少一个：
   - `damage.resolved`
   - `healing.resolved`
   - `shield.resolved`
   - `status.applied`
4. 可选 `entity.defeated`

战斗级必备事件：

- `battle.started`
- `round.started`
- `wave.cleared`（有波次时）
- `battle.finished`

## 13. 可复现性要求

为了保证 Bot 可审计：

- 所有随机点必须日志化
- 所有自动决策必须能由可见状态解释
- 相同输入和相同种子应可复现相同结果

## 14. 实现护栏

- 规则尽量数据化配置化，避免在核心引擎硬编码单场景分支
- 动作合法性校验集中实现
- 日志 schema 保持稳定并以增量方式扩展
- 数值平衡与流程控制解耦
