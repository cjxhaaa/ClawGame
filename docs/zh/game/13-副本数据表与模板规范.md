# 副本数据表与模板规范

## 1. 目标

这份文档把前面的副本、怪物、难度、战斗、装备产出继续推进成“可直接开发的数据表规格”。

目标：

- 把副本系统拆成清晰的表与模板
- 让怪物与波次配置可以数据驱动，而不是写死在代码里
- 让“最多 `6` 个房间”的递进闯关可以完全数据驱动
- 让装备评级奖励和怪物击杀材料掉落都能在表层控制

## 2. 推荐数据分层

副本系统建议拆成六层：

1. 副本定义层
2. 房间配置层
3. 怪物模板层
4. 房间刷怪层
5. Boss 阶段与脚本层
6. 掉落奖励层

对应理解：

- 副本定义层回答“这是什么副本”
- 房间配置层回答“同一个副本的 `1-6` 房怎么变”
- 怪物模板层回答“有哪些怪”
- 房间刷怪层回答“每个房间刷哪些怪”
- Boss 脚本层回答“Boss 在什么时候做什么”
- 掉落奖励层回答“评级给什么装备、击杀给什么材料”

建议新增或细化以下表：

## 4. `dungeon_definitions`

用于定义副本本体。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | text | 副本 ID |
| `region_id` | text | 挂接区域 |
| `name` | text | 对外名称 |
| `public_name_zh` | text | 中文展示名 |
| `tier` | text | `T1-T5` |
| `recommended_level_min` | int | 推荐最低等级 |
| `recommended_level_max` | int | 推荐最高等级 |
| `entry_rank_requirement` | text | 声望阶位门槛 |
| `party_size_min` | int | 最低队伍人数 |
| `party_size_max` | int | 最高队伍人数，当前建议为 `4` |
| `room_count` | int | 当前固定上限为 `6` |
| `round_limit` | int | 每个房间统一回合上限，当前为 `10` |
| `boss_monster_id` | text | 主 Boss 模板 |
| `is_active` | bool | 是否启用 |

## 5. `dungeon_room_configs`

用于定义同一个副本在不同房间下的变化。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | text | 房间配置 ID |
| `dungeon_id` | text | 所属副本 |
| `room_index` | int | 第几房，取值 `1-6` |
| `hp_multiplier` | numeric | 怪物生命倍率 |
| `damage_multiplier` | numeric | 怪物伤害倍率 |
| `heal_shield_multiplier` | numeric | 怪物治疗护盾倍率 |
| `ai_complexity` | text | `basic` / `medium` / `high` |
| `extra_mechanic_flag` | bool | 是否启用额外机制 |
| `rating_if_failed_here` | text | 在该房失败时的默认评级 |
| `timeout_consumes_entry` | bool | 超回合失败是否消耗次数，当前固定 `false` |

| `public_name_zh` | text | 中文展示名 |

| `monster_class` | text | 普通怪 / 精英 / Boss / 召唤物 |
| `base_hp` | int | 基础生命 |
| `base_physical_attack` | int | 基础物攻 |
| `base_magic_attack` | int | 基础魔攻 |
| `base_physical_defense` | int | 基础物防 |
| `base_magic_defense` | int | 基础魔防 |
| `base_speed` | int | 基础速度 |
| `base_healing_power` | int | 基础治疗 |
| `ai_profile` | text | 行为档案 |

## 17. 命名与 ID 规范

建议统一命名方式：

- 副本：`dng_*`
- 房间配置：`drc_*`
- 怪物：`mon_*`
- Boss：`boss_*`
- 房间：`room_*`
- Boss 阶段：`phase_*`
- 评级装备表：`reward_*`
- 材料掉落表：`mat_*`

## 18. 后端实现建议

建议后端模块按以下顺序开发：

1. 先实现 `dungeon_definitions` + `dungeon_room_configs`
2. 再实现 `dungeon_monster_templates` + `dungeon_monster_skills`
3. 然后实现 `dungeon_rooms` + `dungeon_room_monsters`
4. 接着实现 `dungeon_boss_phases` + `dungeon_boss_phase_actions`
5. 最后接 `dungeon_rating_reward_tables` 和 `monster_material_drop_tables`
