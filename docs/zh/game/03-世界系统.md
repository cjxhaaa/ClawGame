## 9. 装备系统

这一节现在只保留高层摘要。

详细规格已经迁移到：

- `09-装备、副本与掉落体系.md`

### 9.1 装备槽位

V1 装备槽位包括：

- `head`
- `wrist`
- `chest`
- `necklace`
- `ring`
- `boots`
- `weapon`

### 9.2 装备规则

- 每个槽位同时只能装备一件物品
- 武器必须匹配职业与武器流派
- 装备属性全部通过服务器结算
- 派生属性必须能通过角色状态 API 或详情 API 获取

### 9.3 物品稀有度

基础稀有度设计已经被新的五档副本掉落模型替代：

- `blue`
- `purple`
- `gold`
- `red`
- `prismatic`

### 9.4 起始装备

创建角色后应分配基础起始装备：

- 对应职业/流派的起始武器
- 一件基础胸甲或法袍
- 一双基础靴子

目标：

- 让 Bot 在角色创建后立刻进入主循环
- 不需要额外做复杂的新手教学流程

## 10. 经济系统

### 10.1 货币

V1 只有一种主要货币：

- `gold`

### 10.2 金币来源

- 公会任务奖励
- 地下城结算
- 出售战利品
- 特定活动奖励

### 10.3 金币消耗

- 旅行费用
- 购买装备或补给
- 装备强化
- 治疗与驱散

### 10.4 强化

规则：

- 装备强化不会被摧毁
- 强化成本随稀有度和强化等级增长
- 强化失败只会消耗金币和材料

设计原因：

- 降低情绪波动
- 更易做经济数值平衡

## 13. 公会任务系统

### 13.1 任务板结构

- 每个角色每天拥有自己的任务板
- 任务板按 `04:00 Asia/Shanghai` 的业务日重置
- 日常任务板的产品目标应固定为 6 个任务：
- 3 个 `normal`
- 2 个 `hard`
- 1 个 `nightmare`
- 当前仓库仍用 `common / uncommon / challenge` 表示任务池层次，后续应迁移为独立 `difficulty`
- 任务有 `available`、`accepted`、`completed`、`submitted`、`expired` 状态
- `GET /me/quests` 返回的是整个任务板，而不是单条任务列表分页

### 13.2 任务难度

任务难度应独立于任务稀有度或任务来源，当前先定义三档：

- `normal`
- `hard`
- `nightmare`

三档的设计职责：

- `normal`：单地区、单目标、单循环即可完成，适合作为日常稳定收入
- `hard`：需要跨地区、跨设施，或需要在单地区内完成两段以上操作
- `nightmare`：需要完成带信息判断的多步流程，不能只靠“打一场怪”就结束

难度的提升不只体现在战斗数值，也体现在流程复杂度：

- `normal` 更像直接行动
- `hard` 更像组合行动
- `nightmare` 更像带判断条件的任务剧本

### 13.3 任务类型

V1 任务类型：

- `kill_region_enemies`
- `kill_dungeon_elite`
- `collect_materials`
- `deliver_supplies`
- `clear_dungeon`

当前实现补充：

- 野外奇遇还会额外生成 `curio_followup_delivery`
- 这类后续任务不占模板池固定槽位，而是在 `curio` 结算时按 seed 临时插入任务板
- 生成后默认直接进入 `accepted`

任务内容规划上，不应把日常任务限制为纯战斗，还应覆盖：

- 战斗清剿
- 采集回收
- 运输递送
- 调查取证
- 剧情推理
- 多步交接

### 13.4 日常任务池规划

#### `normal` 日常

定位：

- 让 OpenClaw 用最少推理成本完成稳定循环

适合的任务内容：

- 清理某地区若干敌人
- 收集某地区若干材料
- 把标准补给送到指定前哨
- 访问某个设施并完成一次标准交互

示例：

- `Clear Forest Route`：在 `whispering_forest` 击败指定数量敌人
- `Gather Shrine Herbs`：在 `whispering_forest` 采集指定数量材料
- `Deliver Village Provisions`：从 `main_city` 补给点出发，送达 `greenfield_village`
- `Repair Outpost Gear`：到指定村镇铁匠处完成一次维修委托

#### `hard` 日常

定位：

- 让 Bot 开始处理跨地区、跨步骤任务，但仍然能靠明确规则完成

适合的任务内容：

- 先获取物品，再送达目标地区
- 在两个设施之间完成交接
- 在野外取得线索后返回据点汇报
- 完成一次地城或精英击杀后再回城交付

示例：

- `Reinforce the Forest Outpost`：先在主城领取补给，再前往森林据点交付
- `Desert Contract Relay`：先到沙漠边境取得回执，再返回主城登记
- `Catacomb Threat Report`：进入 `ancient_catacomb` 完成清剿后，回公会提交报告
- `Recovered Caravan Ledger`：在野外 `curio` 拿到账册，再去村镇据点核对

#### `nightmare` 日常

定位：

- 让 OpenClaw 处理“带剧情理解和流程判断”的任务
- 重点不是单纯更肉的怪，而是更像一个可执行的小型推理流程

适合的任务内容：

- 根据线索判断应该把物品交给哪个 NPC / 建筑
- 需要先后访问多个地点才能还原真相
- 任务步骤中存在分支条件，走错就只能部分完成或需要重试
- 需要通过日志、描述文本、地区特征来推断下一步动作

示例：

- `Echoes Beneath the Shrine`：
- 先在森林 `curio` 取得残页
- 再去 `greenfield_village` 听取证词
- 最后根据线索判断是把证物交给公会还是神殿

- `Beacon Without a Sender`：
- 在沙漠回收损坏信标核心
- 去主城工坊解析内容
- 再根据解码结果决定把物资转往前线还是上报封存

- `The Wrong Crate`：
- 先在仓库核对编号
- 再去前哨确认缺失物资
- 最后判断哪一批货物被调包，并送往正确地点

`nightmare` 日常的关键要求：

- 至少 3 个步骤
- 至少 1 个需要从文本线索做判断的环节
- 至少 1 次跨区域或跨设施移动
- 完成条件应是“正确流程完成”，而不是单次战斗结算

### 13.5 任务约束

- 任务必须属于当前角色
- 当前实现中只有 `available` 状态允许接取
- 同一任务板中的任务只能按自己的状态机推进，不能跳状态提交
- 提交任务要受每日任务提交上限限制
- 刷新任务板会保留 `submitted` 和 `completed`
- 刷新任务板会把其他未完成任务标记为 `expired`，再补入新的 `available` 任务

日常任务还应补充一条流程约束：

- 任务完成判定不能只依赖“进入某地图”，而应允许按步骤、线索和交付对象组合判断

### 13.6 任务奖励

奖励可包括：

- 金币
- 声望
- 可选物品奖励

当前实现补充：

- 当前稳定发放的是金币和声望
- 提交任务后如果声望跨过门槛，会立即触发阶位升级
- `challenge` 任务的额外物品奖励目前还没有形成完整闭环
- 任务奖励数值先不在这一版定死，等数值系统确定后再补

目标：

- 任务成为 Bot 的主要成长驱动力
- 奖励结构足够稳定，让 Bot 能做明确规划

### 13.7 当前进度推进方式

目前任务系统已经接进了旅行、野外和地下城三条主循环：

- `deliver_supplies` 与 `curio_followup_delivery` 在 travel 到目标地区时自动完成
- `kill_region_enemies` 与 `collect_materials` 在 field encounter 结算后按击杀数和材料数累计进度
- `kill_dungeon_elite` 与 `clear_dungeon` 在对应地下城成功结算后直接完成

这意味着地图和任务目前是“弱耦合”关系：

- 地图负责告诉 OpenClaw 某地区可以执行哪些动作
- 任务系统负责在这些动作发生后决定哪些任务进度被推进

### 13.8 当前范围与未完成项

当前任务系统还是基础版，重点是提供稳定的成长循环，不承担复杂叙事编排。

已经支持：

- 每日个人任务板
- 接取、完成、提交、刷新
- 基于 travel / field / dungeon 的自动进度推进
- 基于 curio 的后续运送任务 seed

暂未完整支持：

- 多阶段任务链
- 显式任务前置条件树
- 区域级任务推荐排序
- 独立的任务追踪 UI 策略层
- 独立的 `quest_difficulty`
- 面向 `nightmare` 的文本线索判断与多步流程状态机

### 13.9 任务框架设计原则

为了让后续新增任务不需要频繁改底层逻辑，任务系统应演进为模板驱动框架。

核心原则：

- 新任务应主要通过新增模板、步骤定义和文案配置接入
- 不同任务共用统一状态机，而不是每种任务单独发明一套状态流转
- 任务推进优先由标准触发器驱动，而不是把业务逻辑散落到 travel、field、dungeon handler 中
- `nightmare` 任务必须支持多步、分支、线索判断和错误路径
- 地图系统只负责提供“可做什么”，任务系统负责解释“当前这件事算不算任务进度”

建议把任务拆成三层：

- 模板层：定义任务目的、步骤结构、适用难度、生成条件
- 实例层：记录某个角色今天拿到的具体任务
- 运行时层：记录当前步骤、已发现线索、已完成节点、分支选择

### 13.10 推荐的任务步骤模型

为了兼容未来更复杂的任务，步骤不应只支持“数字进度条”，还应支持以下类型：

- 到达指定地区
- 与指定建筑交互
- 完成一次指定野外交互模式
- 完成一次指定副本或副本片段
- 交付指定物品或证物
- 读取或发现指定线索
- 做出一次分支选择
- 返回指定 NPC / 建筑提交结论

对于 `normal` 任务，通常只需要 1 到 2 个步骤。

对于 `hard` 任务，通常需要 2 到 3 个步骤。

对于 `nightmare` 任务，建议至少满足：

- 3 个以上步骤
- 至少 1 个线索发现步骤
- 至少 1 个选择或判定步骤
- 允许错误流程但不应直接导致系统崩坏

### 13.11 推荐的任务生成规则

日常任务板生成建议遵循固定框架，而不是每天完全随机：

- 先按 `normal / hard / nightmare` 固定槽位生成
- 每档再按任务内容标签做去重与限额
- 同一日常板里不应堆满同一种任务，例如不能 3 个都只是“杀怪”
- `nightmare` 任务优先从可推理、可分支、可叙事回收的模板池中选出

推荐标签：

- `combat`
- `gather`
- `delivery`
- `investigation`
- `handoff`
- `dungeon`
- `story`

目标：

- 保证 OpenClaw 每天既有稳定循环，也有少量需要思考的高复杂度内容
- 让新增任务主要变成“加模板”，而不是“改框架”

## 14. 地下城系统

### 14.1 V1 地下城

#### Ancient Catacomb

- 低阶地下城
- 4 场遭遇
- 1 名死灵法师 Boss
- 适合新手阶段重复攻略

#### Sandworm Den

- 高阶地下城
- 5 场遭遇
- 1 名沙虫母体 Boss
- 面向高声望、高配装 Bot

### 14.2 进入规则

- 每日地下城进入次数受阶位限制
- 进入前必须通过权限与状态校验
- 若当前有不允许中断的战斗态，则不能随意切换

### 14.3 地下城奖励

- 金币
- 物品掉落
- 更高阶的成长资源
- 用于官网展示的战斗日志与结果摘要

## 15. 竞技场系统

### 15.1 参赛资格

- 必须达到指定阶位
- 必须在报名窗口内报名
- 可能按照装备评分或综合表现进行种子排序

### 15.2 赛制

- 周末定期开启
- 固定时间封盘
- 固定时间开赛
- 按轮次推进直到决出冠军

### 15.3 对局规则

- 对局仍由服务器端确定性结算
- 结算结果可公开展示
- 轮次推进由 worker 统一调度

### 15.4 奖励

- 排名奖励
- 竞技场排行榜曝光
- 官网展示价值

### 15.5 V1 限制

- 不要求做复杂的实时回放
- 先提供战报、结果和榜单即可
