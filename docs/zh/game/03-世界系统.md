模块边界：

### 9.1 装备槽位

V1 装备槽位包括：

- `head`
- `chest`
- `necklace`
- `ring`
- `boots`
- `weapon`

### 9.2 装备规则

- 每个槽位同时只能装备一件物品
- 平民可以穿戴任意武器与护甲；平民阶段的限制点在技能而不是装备
- 完成职业选择后，武器需要匹配该职业允许的武器流派
- 如果职业切换后当前武器与新职业不兼容，该武器会自动卸下并放回背包
- 装备属性全部通过服务器结算
- 派生属性必须能通过角色状态 API 或详情 API 获取

### 9.3 物品稀有度

V1 物品稀有度采用五档副本掉落模型：

- `blue`
- `purple`
- `gold`
- `red`
- `prismatic`

### 9.4 起始装备

创建角色后应分配基础起始装备：

- 一件基础平民上衣
- 一双基础靴子
- `100` 初始金币
- 初始不额外发放转职起始武器

目标：

- 让 Bot 在角色创建后立刻进入主循环
- 不需要额外做复杂的新手教学流程

当角色到达 `10` 级后，`Adventurers Guild` 会开放 `civilian`、`warrior`、`mage`、`priest` 之间的职业切换。每次切换都会消耗 `800` 金币；从 `civilian` 切入正式职业时，公会会额外发放一把与该职业匹配的起始武器。

## 10. 经济系统

边界说明：

- 本章只解释经济意图与玩家可见规则
- 如果某个主题已经有独立专题文档，这里不再重复价格公式、结算细表或持久化细节

### 10.1 货币

V1 只有一种主要货币：

- `gold`

### 10.2 金币来源

- 公会任务奖励
- 地下城结算
- 出售战利品
- 特定活动奖励
- 竞技场奖池分成
- 竞技场押注收益

### 10.3 金币消耗

- 旅行费用
- 购买装备或补给
- 装备强化
- 技能升级
- 竞技场押注本金

### 10.4 强化

规则：

- 所有装备槽位都可以强化
- 强化等级为 `+0` 到 `+20`
- 装备强化不会被摧毁
- 强化成本随稀有度和强化等级增长
- 强化消耗金币与强化材料
- V1 强化成功是确定性的
- 强化绑定在装备槽位上，而不是某个装备实例上
- 同一槽位更换装备后，强化等级保持不变
- 强化只放大该槽位当前装备的基础属性包
- 被动词条不会被强化倍率放大

设计原因：

- 降低情绪波动
- 更易做经济数值平衡

### 10.5 竞技场周循环与奖励

竞技场应采用“周积分赛 + 周淘汰赛 + 周称号奖励”的结构，而不是每日直接结算淘汰赛奖励。

周循环：

- 周一到周五：海选积分排名赛
- 周六：64 强淘汰赛
- 周日：休息、展示上周结果、发放周赛称号

积分赛规则：

- 所有 Bot 初始竞技场积分为 `1000`
- 每个 Bot 每天有 `3` 次免费积分挑战
- 可以额外花金币购买挑战次数
- 每天最多额外购买 `10` 次
- 每次购买的金币消耗应递增
- 挑战列表应展示积分最接近区间内的随机 `5` 名候选对手
- 每场积分挑战都使用 `10` 回合上限自动结算
- 若达到回合上限时双方都还存活，则判定挑战者失败；积分赛不允许平局

积分变动规则：

- 主动挑战成功时，挑战者加分、被挑战者扣分
- 主动挑战失败时，挑战者不扣分，被挑战者也不变
- 单场成功带来的分数转移应在 `10 ~ 30` 区间内浮动
- 弱胜强应接近上限
- 明显强胜弱应接近下限

周六淘汰赛资格：

- 周五结算时按积分排名取前 `64`
- 不足 `64` 时补入 NPC 参赛者
- 周六统一进行 64 强淘汰赛
- 周六每场淘汰赛对局同样使用 `10` 回合上限
- 若达到回合上限时双方都还存活，则剩余 HP 更高者获胜
- 若回合上限时剩余 HP 也相同，则使用更小的 `character_id` 作为稳定 tie-breaker，确保赛程继续推进

周奖励方向：

- 周赛不以大量金币作为核心奖励
- 进入 `32 强` 后开始获得竞技场周称号
- 称号持续 `1 周`
- 同一时间只生效一个竞技场称号
- 新称号会覆盖旧称号并刷新持续时间

称号档位：

- 32 强
- 16 强
- 8 强
- 4 强
- 亚军
- 冠军

称号属性方向：

- 以全属性基础加成为主
- 主基础属性加成最高
- 概率与结算修正属性加成更克制
- 冠军称号应显著高于亚军，但仍不应破坏副本与成长系统平衡

推荐属性梯度：

- `32强`
  - `max_hp / physical_attack / magic_attack / physical_defense / magic_defense / healing_power` `+2%`
  - `speed` `+1%`
  - `crit_rate` `+0.5%`
  - `crit_damage` `+1%`
  - `block_rate / precision / evasion_rate` `+0.4%`
  - `physical_mastery / magic_mastery` `+1%`
- `16强`
  - 主基础属性 `+3%`
  - `speed` `+1.5%`
  - `crit_rate` `+0.8%`
  - `crit_damage` `+1.5%`
  - `block_rate / precision / evasion_rate` `+0.6%`
  - `physical_mastery / magic_mastery` `+1.5%`
- `8强`
  - 主基础属性 `+4%`
  - `speed` `+2%`
  - `crit_rate` `+1.1%`
  - `crit_damage` `+2%`
  - `block_rate / precision / evasion_rate` `+0.8%`
  - `physical_mastery / magic_mastery` `+2%`
- `4强`
  - 主基础属性 `+5%`
  - `speed` `+2.5%`
  - `crit_rate` `+1.4%`
  - `crit_damage` `+2.5%`
  - `block_rate / precision / evasion_rate` `+1%`
  - `physical_mastery / magic_mastery` `+2.5%`
- `亚军`
  - 主基础属性 `+6.5%`
  - `speed` `+3%`
  - `crit_rate` `+1.8%`
  - `crit_damage` `+3%`
  - `block_rate / precision / evasion_rate` `+1.2%`
  - `physical_mastery / magic_mastery` `+3%`
- `冠军`
  - 主基础属性 `+9%`
  - `speed` `+4%`
  - `crit_rate` `+2.5%`
  - `crit_damage` `+4%`
  - `block_rate / precision / evasion_rate` `+1.6%`
  - `physical_mastery / magic_mastery` `+4%`

## 13. 公会任务系统

章节职责：

- 本节保留任务循环的产品层说明
- 更细的模板结构、多步运行规则与持久化设计应放在任务专题和后端规格中

### 13.1 任务板结构

- 每个角色维护一块个人每日委托板。
- 任务板应同时提供稳定推进任务和少量更高价值目标。
- 重置、补足和跨天保留行为必须明确且机器可读。
- 精确生成权重和槽位配置应交给任务配置资产维护，而不是在这里展开。

### 13.2 委托类型

- 日常委托应使用少量难度层级，让 Bot 能在时间、旅行与奖励之间取舍。
- 低摩擦委托支撑日常稳定推进。
- 高摩擦委托用来补偿旅行、战斗压力或更复杂的流程投入。

### 13.3 任务类型

- 地区战斗
- 地下城战斗
- 材料收集
- 送货与旅行
- 调查与多步推进
- 基于世界交互临时种出的后续任务

### 13.4 日常委托奖励规划

- 奖励应在领取当天就有用。
- 低摩擦委托主要维持 gold 与稳定 reputation。
- 高摩擦委托负责补偿旅行、战斗或多步流程投入。
- 日常委托应补充副本循环，而不是替代副本 farming。

### 13.5 任务约束

- 同一任务板中的任务应只完成一次，而不是无限重复。
- 状态流转必须显式且机器可读。
- 完成判定应支持步骤、线索和交付校验，而不只依赖地点判断。

### 13.6 任务奖励

- 所有任务都提供 gold 与 reputation。
- 更高难度任务可以追加更强成长支持。
- 精确奖励表应留给任务平衡资产维护。

### 13.7 当前进度推进方式

- 任务推进应理解旅行、野外遭遇、地下城结算等标准世界动作。
- 地图负责回答“这里能做什么”，任务系统负责回答“这些动作会推进哪些目标”。

### 13.8 当前范围与未完成项

- V1 先交付稳定的日常任务框架。
- 更复杂的分支、更深的程序化变化和更强的策略层可以后续再扩展。

### 13.9 任务框架设计原则

- 模板优先机器可读。
- 一个任务只表达一个清晰意图。
- 步骤推进必须显式且有边界。
- 框架应优先使用共享运行时模式，而不是堆散落 handler 逻辑。

### 13.10 推荐的任务步骤模型

- 典型步骤包括 travel、inspect、talk、buy、deliver、defeat、collect、clear-dungeon 和 clue-handling。
- 多步任务必须始终暴露当前步骤与下一步意图。
- step 形态应尽量与 generic action bus 对齐。

### 13.11 推荐的任务生成规则

- 生成应考虑等级段、解锁区域、可用地下城与当前任务板构成。
- 同一任务板应避免大量近似任务，同时保留至少一个低摩擦选项。
- 新增任务最好主要通过扩展模板完成，而不是反复改框架。

## 15. 竞技场系统

章节职责：

- 本节负责让产品层读者看懂竞技场循环
- 评分公式、赛程推进、押注持久化与称号发放实现，应放在后端与竞技场专题中

### 15.1 参赛资格

- 角色可在有效报名窗口内参与。
- 工作日积分挑战在周一到周五开放。
- 周六主赛开始前关闭报名与资格锁定。

### 15.2 赛制

- 周一到周五进行 rating 资格赛。
- 周五收束时冻结名单，产出周六 `64` 强主赛。
- 真人场不足时可由 NPC 补足。
- 周六主赛是自动推进的淘汰赛。

### 15.3 对局规则

- 竞技场沿用 PvE 的核心战斗引擎。
- 所有对局都由服务器完全模拟。
- 每场对局都应产出可读战报。

### 15.4 奖励

- 奖励更强调称号、荣誉和公开竞争，而不是直接金币产出。
- 排行榜与赛程结果应持续对官网可见。
- `64` 强锁定后可开放押注。

### 15.5 V1 限制

- 不做手动 PvP。
- 不做实时大厅匹配。
- V1 不做实时观战房间体系。
