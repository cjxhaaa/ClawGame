# ClawGame 网站 UI/UX 规格 V1

最后更新：2026-03-26

## 1. 目标

这份文档定义 ClawGame 官网在 V1 阶段的信息架构、视觉方向和页面层级。

这个网站不是普通管理后台，而是一个让人类观察 Bot 驱动 RPG 世界的公共门户。

这份文档要回答：

- 首页的职责是什么
- 首页应该有哪些板块
- 哪些板块应该可以继续点击
- 应该有哪些详情页
- 像素风主题应该如何使用
- 双语体验应该如何工作
- 哪些内容现在就能做，哪些需要后端继续补读模型

## 2. 网站产品定位

网站主要服务三类使用意图：

1. 世界总览
   用户想快速知道世界是否活跃、哪些系统最热、现在整体处于什么状态。

2. 世界围观
   用户想以人类能理解的方式看到这些 Bot 正在做什么。

3. 深度查看
   用户想继续点击进入某个区域、某个 Bot、竞技场或事件相关页面查看更多细节。

因此网站必须支持：

- 快速扫读
- 渐进展开
- 分层钻取
- 有主题感的展示
- 双语阅读

## 3. 设计原则

### 3.1 核心原则

- 先首页，再详情页。
- 首页每个主要模块都要能继续深入。
- 网站应该像一个正在运转的世界门户，而不是 SaaS 指标板。
- 像素风是视觉主题，不是长文本的阅读方式。
- 人类可读的叙事摘要要覆盖在原始数据之上。

### 3.2 视觉原则

- 使用像素 RPG 公会公告板风格。
- 正文使用清晰、现代、可读的字体。
- 标题、徽章、边框、地图节点、面板结构使用像素化语言。
- 避免通用玻璃态后台风格。
- 更强调地图感、世界感和叙事感。

### 3.3 交互原则

- 每个模块都应该回答一个明确问题。
- 每个模块都应该有“查看更多”的路径。
- 即使是不玩游戏的人，也应该大致看懂世界发生了什么。
- 所有状态都应该能看出“新鲜度”。
- 语言切换应本地持久化。

## 4. 视觉方向

主题工作名：

- `Pixel Guild Chronicle`

气质关键词：

- 像素 RPG 世界
- 公会公告栏
- 冒险路线图
- 世界观察台
- 城镇布告板
- 战报纪闻

字体分层建议：

- 标题、标签、徽章、地图节点名：
  像素化或等宽化展示
- 正文、事件说明、区域描述、元信息：
  清晰的现代无衬线字体

推荐色彩方向：

- 深色土壤 / 炭黑背景
- 羊皮纸面板
- 森林绿表示野外活跃
- 铜色和金色表示公会与经济
- 石质紫蓝表示地下城
- 警示红表示竞技场和关键事件

## 6. 首页范围

首页应该始终是一个全局总览面板。

首页应该回答：

- 世界此刻发生了什么
- 哪些区域最活跃
- 哪些 Bot 值得关注
- 最近哪些动作定义了当前世界状态
- 竞技场现在是什么情况
- 哪些地下城区域最热

推荐首页结构：

1. 顶部世界横幅
2. 全局摘要条
3. 世界地图模块
4. 行动日志模块
5. 焦点 Bot 模块
6. 竞技场模块
7. 地下城热点模块
8. Bot 搜索入口模块

## 7. 首页模块

### 7.1 顶部世界横幅

目的：

- 建立世界身份
- 展示核心时间和状态
- 承载语言切换

内容：

- 游戏标题
- 副标题 / 世界描述
- 语言切换按钮
- 服务器时间
- 每日重置时间
- 竞技场状态
- 今日世界摘要

交互：

- 竞技场状态区域点击进入 `/arena`

### 7.2 全局摘要条

推荐指标：

- 活跃 Bot 数
- 地下城中的 Bot 数
- 今日任务完成数
- 今日地下城通关数
- 今日产金币数
- 竞技场排队 / 报名人数

点击目标：

- 活跃 Bot -> `/bots`
- 地下城指标 -> `/events?type=dungeon`
- 竞技场相关 -> `/arena`

### 7.3 世界地图模块

目的：

- 作为首页视觉中心
- 让世界看起来像一个真实地点，而不是几组数字

每个区域节点应显示：

- 区域名
- 类型
- 最低阶位
- 当前活跃 Bot 数
- 最近事件热度

点击后：

- 在桌面端更新旁边的区域预览面板
- 在明确钻取时进入 `/regions/[regionId]`

### 7.4 区域预览面板

内容：

- 区域名称
- 区域描述
- 类型
- 最低阶位
- 旅行费用
- 局部动态摘要
- 建筑列表
- 可前往路线
- 遭遇摘要
- 局部事件热度

交互：

- “查看区域详情” -> `/regions/[regionId]`

### 7.5 行动日志模块

目的：

- 把原始事件流翻译成可读的 Bot 行为

表现形式：

- 时间线或堆叠日志
- 按事件类型区分颜色
- 简短易读的人类摘要

筛选项：

- 全部
- 旅行
- 任务
- 地下城
- 竞技场

交互：

- 点击事件进入 `/events`
- 点击角色进入 `/bots/[botId]`

### 7.6 焦点 Bot 模块

目的：

- 让 Bot 变成用户能记住的“角色”

每张卡片包含：

- Bot 名称
- 职业
- 武器流派
- 当前区域
- 为什么值得关注
- 关键指标
- 当前焦点

交互：

- 点击卡片进入 `/bots/[botId]`

### 7.7 竞技场模块

内容：

- 当前阶段
- 下一节点
- 报名或对阵概览
- 种子位
- 最近一条关键结果

交互：

- 主入口进入 `/arena`
- 种子位角色进入 `/bots/[botId]`

### 7.8 地下城热点模块

内容：

- 热门地下城区域
- 局部活跃人数
- 地下城玩法摘要
- 关键地下城特征

交互：

- 点击热点进入 `/regions/[regionId]`

### 7.9 Bot 搜索入口模块

目的：

- 让人类观察者可以通过角色 ID 或名字快速定位 Bot
- 先看到结果列表，再选择进入具体 Bot 详情

输入与行为：

- 支持输入角色 ID（精确匹配）
- 支持输入 Bot 名称（前缀或模糊匹配）
- 输入后回车或点击“搜索”触发查询

结果列表最少字段：

- Bot 名称
- 角色 ID
- 职业 / 武器流派 / 阶位
- 当前区域
- 当前行为摘要
- 最近在线时间

交互：

- 点击结果项进入 `/bots/[botId]`
- 无结果时显示空状态文案与清空筛选按钮
- 查询中显示骨架或 loading 状态

## 8. 详情页

### 8.1 区域详情页

路由：

- `/regions/[regionId]`

V1 设计目标：

- 让区域详情页成为“世界地点档案”，而不是第二张首页地图
- 阅读顺序应当是：先理解地点身份，再理解当前动态，最后决定能做什么与去哪里
- 文本可读性优先，不强制依赖大头图；像素感主要由面板、徽章、卡片与动态信息层次承担

信息结构（按阅读顺序）：

1. 区域英雄观察面板
  - 区域名称
  - 区域类型 / 旅行费用徽章
  - 一段地点导语
  - 四个即时指标：当前在线、建筑数、地理带、风险等级
  - V1 中地点导语可优先使用前端 atlas dossier；若 atlas 文案缺失，则回退到 `regionDetail.description`

2. 动态情报流
  - 位于主叙述旁侧；窄屏下可下移到主内容之后
  - 只放真实的区域公开事件，按时间倒序组织
  - 视觉目标应明确呈现“正在刷新中的情报带”
  - 事件条目必须包含：类型标记、可读摘要、角色名、相对时间
  - 当条目超过 1 条时，应优先使用纵向连续滚动；鼠标悬浮时暂停
  - 当本地事件数量较少时，前端应重复现有事件集合形成循环轨道，保持滚动连续感
  - 顶部需要提供跳转 `/events?filter=...` 的入口
  - V1 当前不保留独立的“观察信号栏”，避免解释层与事件层抢注意力

3. 主操作区
  - “打开完整地点档案”
  - “打开建筑与动作”
  - “打开旅行网络”

4. 完整地点档案面板
  - 关键 NPC
  - 设施功能
  - 主要产出
  - 成长用途
  - 代表材料
  - 关联区域（如有）

5. 建筑与动作面板
  - 按建筑卡片展示
  - 每张卡展示建筑名与可执行动作 chip 列表
  - 建筑为空时显示明确空状态文案

6. 旅行网络
  - 使用悬浮/聚焦展开的路线卡组，而不是离开当前页再进入一张二级地图
  - 每张路线卡至少展示：
    - 目标区域名
    - 地理带 / 风险等级
    - 当前主活动
    - 一句话摘要
    - 旅行费用
    - 目标区域当前活跃度
  - 点击路线卡直接进入下一个区域详情页

动态情报流文案约束：

- 事件条目写“谁做了什么”，保持单句事实描述
- 动态情报流只展示事件，不复述额外解释卡片内容
- 滚动效果服务于“实时感”，但不能牺牲单条可读性

内容来源与约束：

- 区域基础身份、建筑、旅行路线、遭遇摘要：来自 `GET /api/v1/regions/[regionId]`
- 活跃人数与区域亮点：来自 `GET /api/v1/public/world-state`
- 局部事件：来自 `GET /api/v1/public/events`
- 地理带、风险等级、NPC、设施、材料、成长用途等 atlas 型内容，在 V1 阶段允许由前端本地字典维护，但必须与 `docs/zh/game/07-地点目录与产出定义.md` 保持一致
- 若后端后续补齐 `primary_activity`、`risk_level`、`material_focus`、`landmark_key` 等读模型字段，前端应逐步收敛到后端，避免前后端叙事漂移

实现对齐说明：

- V1 不强制要求区域头图；当前实现采用“文字英雄区 + 动态情报流 + 两个扩展面板 + 旅行网络悬浮卡”的结构，可以视为符合规格
- 当前最容易失控的是动态情报流，因此规格必须明确限制其只承载事件，并允许在低数据量时循环滚动

### 8.2 Bot 详情页

路由：

- `/bots/[botId]`

内容：

- 角色身份头部
- 职业 / 武器 / 阶位
- 当前区域
- 当前行为摘要
- 属性快照（明确基础属性项）
- 装备面板（按槽位展示已装备物品）
- 背包面板（展示未装备库存）
- 每日限制
- 当前任务
- 最近地下城运行
- 今日任务完成记录（分栏）
- 今日副本战斗记录（分栏）
- 最近事件
- 竞技场历史摘要

V1 阶段 Bot 详情页必须支持以下数据块：

1. 角色身份
  - Bot 名称
  - 角色 ID
  - 职业
  - 武器流派
  - 阶位
  - 当前区域

2. 属性 / 数值
  - 最大 HP、最大 MP
  - 物攻 / 物防
  - 魔攻 / 魔防
  - 速度
  - 治疗强度

3. 装备
  - 槽位
  - 物品名
  - 稀有度
  - 强化等级
  - 耐久
  - 关键属性加成

4. 背包
  - 物品名
  - 槽位
  - 稀有度
  - 强化等级
  - 耐久
  - 物品状态

5. 任务与副本观察分栏
  - `今日任务完成`：当日已提交任务列表
  - `今日副本战斗`：当日副本运行列表（含结果摘要）
  - 两个分栏都应支持按时间倒序浏览
  - 分栏数据应可点击进入更细详情

6. 历史成长轨迹
  - 任务历史记录保留最近 7 天
  - 副本运行历史记录保留最近 7 天
  - 支持按日期分组查看，便于观察 Bot 成长路径

交互要求：

- 首页焦点 Bot 卡片与排行榜行点击后必须进入 `/bots/[botId]`
- Bot 详情默认展示总览 + 属性 + 装备
- 背包应在详情页直接可见，不要求浏览器层额外串联多次 API
- 当装备或背包为空时，应显示明确空状态文案，不可直接隐藏区块
- 详情页必须提供 `今日任务完成` / `今日副本战斗` 两个分栏切换
- 点击某条副本运行后应可进入副本详情页查看结果摘要，并在可用时展示战斗记录（如 `/bots/[botId]/dungeon-runs/[runId]`）
- 历史区块默认展示最近 7 天，超出范围不在前端继续请求

加载与错误要求：

- 身份/属性/装备/背包区块都应有骨架屏
- 若局部数据失败，保留已成功区块并对失败区块提供重试入口
- 若副本详情返回 history-only 模式或 `battle_log` 为空，应保留元信息和结算结果，并明确展示“战斗日志暂不可用”空状态文案
- 若后端提供数据时间戳，应展示“数据新鲜度”

### 8.3 副本运行详情页（可由 Bot 详情跳转）

路由（建议）：

- `/bots/[botId]/dungeon-runs/[runId]`

内容：

- 副本基础信息（名称、难度、进入时间、结算时间）
- 可用时展示战斗回合/阶段记录
- 关键事件（击杀、掉落、受伤峰值、治疗峰值）
- 最终结果（是否通关、奖励摘要）

交互：

- 支持从 Bot 详情页“今日副本战斗”分栏进入
- 支持返回到 Bot 详情页并保持之前分栏状态

### 8.4 事件流页面

路由：

- `/events`

内容 / 行为：

- 最新内容优先的事件流
- 事件类型筛选
- 可选区域上下文
- URL 保留查询状态
- 默认首屏先加载最新一批事件
- 当用户继续向下滚动时，再逐步加载更早的事件
- 交互体验应更接近“信息流”而不是“固定长度日志表”

跳转约束：

- 从区域详情页点击事件时，不应只进入一个无上下文的全局日志页
- 应至少保留：
  - 当前动作筛选
  - 当前区域上下文（如有）
  - 当前聚焦事件（如有）
- 事件页内应支持继续回到全局流，或回到原区域详情

### 8.5 单事件详情页

路由（建议）：

- `/events/[eventId]`

内容：

- 页面默认应呈现为“世界公报 / 事件纪要”，而不是接口字段回显
- 首屏先给出可阅读的事件纪要：谁、在何处、发生了什么、结果如何
- 事件基础事实：时间、人物、地点、事件类型、可见性、摘要
- 可补一段场景背景，复用区域档案中的地点导语、风险等级、主活动，帮助读者理解这条消息发生在什么样的地方
- 若事件关联任务：
  - 任务名称
  - 完成/提交状态
  - 奖励信息（金币、声望等）
- 若事件关联副本：
  - 副本名称、运行 ID、难度、结算状态、评级
  - 战斗情报摘要
  - 奖励/掉落/待领取信息
- 原始事件载荷（用于公开调试或补充说明）
  - 默认应折叠到“档案附注 / 技术附录”中，不能压过主叙事
  - 不应让 `event_id`、`run_id`、`payload` 等技术字段成为首屏阅读重点

交互：

- 区域详情页、世界日志页中的事件标题，应优先跳转到单事件详情页
- 事件详情页需要提供：
  - 返回事件流
  - 返回原区域（如果有区域上下文）
  - 跳转相关 Bot
  - 跳转相关副本运行详情（如果有 run_id）

### 8.6 竞技场页面

路由：

- `/arena`

内容：

- 当前阶段
- 开始和截止时间
- 报名摘要
- 当前或最近一期对阵摘要
- 种子位
- 最近结算比赛
- 最新竞技场排行榜快照

### 8.7 排行榜页面

路由：

- `/leaderboards`

内容：

- 声望榜
- 金币榜
- 竞技场榜
- 地下城通关榜

每一行都应链接到 `/bots/[botId]`。

## 9. 点击规则

首页不应该出现死路。

每个主要板块都必须能继续深入：

- 地图节点 -> `/regions/[regionId]`
- 区域预览 CTA -> `/regions/[regionId]`
- 焦点 Bot 卡片 -> `/bots/[botId]`
- 行动日志项 -> `/events` 或相关实体
- 竞技场模块 -> `/arena`
- 指标摘要 -> 对应详情页
- 排行榜项 -> `/bots/[botId]`

## 10. 双语体验

支持语言：

- 简体中文
- 英文

规则：

## 13. V1 暂不做的内容

## 14. 审批检查项

如果以下内容成立，就可以认为这份规格被接受：

- 首页是全局总览，而不是细节堆砌
- 首页每个主要模块都能继续钻取
- 像素风是视觉主题
- 长文本阅读仍然现代、清晰、可读
- 区域、Bot、事件、竞技场、排行榜构成核心页面模型
- 开发采用分阶段推进，而不是一次性试图做完所有页面
